> 日期：2026-07-09
> 方法：按「流程覆盖 / 系统机制 / 度量闭环 / 人与外部」四维列候选盲点，逐一到代码与模板中验证（区分真盲区 / 部分覆盖 / 已覆盖），按 发生概率 × 伤害 排序。

## 🔴 高危盲点（量产工厂的日常就会撞上）

### 1. 让步接收 / 临时代料没有任何流程载体 ✅ 已实施（2026-07-10）

> 落地：产品轴新增量产让步单，覆盖偏差/影响/围堵、批次/数量/期限、受影响料号、独立责任人与批准人、受控证据；编制人禁止自批。批准单到期后自动转为 expired，只能关闭或转入同产品 ECO；草稿只能取消，待审批单不能绕过审批直接闭环。

**现状**：全库搜「让步/代料/偏差/concession/deviation」，只在 gateStandard 的 exceptionStrategy 文字里出现（"良率/交付偏差进入管理层周会跟踪"）。产线上"这批料先用替代料顶一下""这个尺寸超差但能装，先出货"是工厂**高频事件**，目前系统里无处记录。

**为什么危险**：这类临时决定绕过 ECO（ECO 是正式项目，起不了"今天下午就要决定"的作用），会在体外积累成隐形变更——三个月后没人记得哪批货用了什么代料。追溯断裂 + 变更管理被掏空。

**最小闭环**：一张「让步接收单」（模板族 F01 变体：偏差描述/影响评估/批准人/限定批次或期限/是否转正式 ECO），挂在产品或项目下的轻量记录表——不需要新项目类型，需要的是**比 ECO 轻得多的登记+批准+到期强制复核**。到期未转 ECO 自动提醒。

### 2. 认证证书生命周期无人监管 ✅ 已实施（2026-07-10）

> 落地：证书台账增加续期责任人、续期状态、续期说明和替代证书；覆盖校验基准日取“今天/项目目标日”较晚者。自动化在到期前 90/30 天提醒并以证书+有效期+窗口去重，到期后自动失效；替代证书必须属于同一产品。

**现状**：全库无任何证书有效期字段（`grep 有效期|expir|validUntil` 零命中于 schema 与自动化规则）；证书是普通上传文件（fileType=certification）。量产发布硬闸口检查 UN38.3/MSDS/电芯认证**存在且审核通过**，但不检查**有效期是否覆盖量产周期**；12 条自动化规则里没有证书到期提醒。

**为什么危险**：UN38.3、CB、安规证书都有有效期/标准版本迭代。项目关闭后进入量产的产品，证书到期 = 出货违规，而此时已经没有任何项目在盯它——这正是"项目制管理"对"持续合规"的结构性盲区。

**最小闭环**：① 证书类文件加结构化字段（编号/机构/签发/到期/适用范围）——F10 归档单模板已有这些列，缺的是入库；② 一条自动化规则：到期前 90/30 天推 cert/battery_safety；③ 发布预检加一项"证书有效期 ≥ 预计量产窗口"。

### 3. 项目只有成功路径——没有「终止」语义 ✅ 已完整实施（2026-07-10）

> 落地：终止前必须完成七项结构化善后（模具、物料、样品、客户承诺、财务合同、IP 文件、知识沉淀），逐项填写处置方案和证据，并由独立批准人批准；未批准或任一项未完成时服务端拒绝终止。组合看板新增“归档 / 终止”视图，区分正常关闭与中途终止并显示理由和日期。

**现状**：Gate 结论只有 approved / conditional / rejected（schema.ts:1062），rejected = 停留整改重审（CEH-16 拍板），**不是终止**。项目表只有 `archived` 布尔。没有 kill 决策、没有终止评审、没有善后清单。

**为什么危险**：立项后发现商业不成立/技术不可行的项目**杀不掉**，只能烂在看板上变僵尸项目（占资源统计、拉低组合健康度）；更实际的是善后无载体——已开的模具、已购的长交期物料、已承诺客户的样品，终止时的处理决定无处记录。IPD 的 Gate 本义是三向决策：Go / Redirect / **Kill**，现在缺了第三向。

**最小闭环**：① 项目加「终止」动作（权限同量产发布，理由必填）；② 终止走一张「终止评审记录」（F01 变体：终止原因/沉没成本盘点/模具物料样品处置/客户沟通/知识沉淀）；③ 组合看板把 terminated 与 archived 区分展示。顺带补「暂停/挂起」状态（暂停项目不进逾期统计）。

### 4. 「有条件通过」的条件项是自由文本，无人盯关闭 ✅ 已实施（2026-07-10）

> 落地：Gate 评审支持一次录入多条结构化条件，每条独立责任人、期限、条件项和行动项；任何未关闭条件都会硬卡后续 Gate。Release follow-up 与历史条件统一进入同一条件机制，不再保留自由文本侧门。

**现状**：gate_reviews.conditions 是一个 **text 字段**（schema.ts:1182）——评审写"有条件通过：①模具比价回签 ②验证矩阵会签"，这两条不落任务、不落责任人期限、无到期提醒（自动化只有 gate_decision_notify 播报一次）。conditional 强制发布的 followUp 有留痕，但同样无跟踪规则。

**为什么危险**：例外通道是流程最容易溃烂的地方。条件项写完即忘 → "有条件通过"事实上等于"通过" → 下一个人有样学样，Gate 纪律从这里失守。你们把发布硬闸口修得很严，但侧门是开的。

**最小闭环**：① 条件项结构化：评审时逐条录入（描述/责任人/期限），自动生成跟踪任务或复用 gate blocker 机制；② 自动化规则：条件项到期未关闭 → 推 PM+责任人，且下一 Gate 就绪度把"上一 Gate 条件项未关闭"列为硬维度；③ 发布 followUp 同样进到期扫描。

### 5. 量产后治理真空：NPD 无关闭 Gate + 量产变更靠自觉 ✅ 已实施（2026-07-10）

> 落地：六轨均拆分 Release Gate 与稳定期 Close Gate；Close 依赖两期/14 天稳定记录、证书覆盖、条件闭环和正式移交接收。关闭后产品进入 maintenance、项目归档，售后、ECO、软件发版和 EOL 留在产品轴。

**现状**：ECO 有 em4 关闭、DRV 有 dm3 关闭，但 **NPD/JDM 的 MP 阶段是"持续"**，项目没有明确终点——何时 archived 靠 PM 手感。同时瘦身计划删掉了 mp4"工程变更管理"常设任务（正确），但量产后变更入口只剩"自觉走 ECO"，与盲点 1 叠加后，量产期的变更/证书/售后实际处于三不管。

**最小闭环**：给 NPD/JDM 补一个轻量「量产移交/项目关闭」Gate（爬坡达标 + 移交运营清单：证书台账/售后通道/变更入口确认），关闭后产品责任正式移交产品维护（产品库 Revision 轴），项目归档。这样"项目制"与"持续运营"有了明确交接棒。

## 🟡 中风险盲点（值得排期，不急于本月）

### 6. 选错轨道无纠偏通道（转轨） ✅ 已实施（2026-07-10）

受控转轨采用“关旧开新”，新项目按目标模板生成，迁移成员、未结问题和文件引用并保留来源 ID；旧项目以转轨理由终止归档，两个项目都写入追溯日志。目标轨道入口硬卡继续生效，禁止直接修改 category。
exceptionStrategy 到处写"转入 DRV/NPD 评审"，但系统没有转轨操作：改 category 会让已埋任务与新模板错位，实际只能手工建新项目、旧项目弃置，交付物/问题/评审记录不迁移。建议：转轨向导（关旧开新 + 引用式迁移交付物与 issue + 活动日志链接两个项目）。

### 7. 项目费用无结构化跟踪 ✅ 已实施（2026-07-10）

> 落地：项目费用表覆盖模具、认证、NRE、打样、差旅和其他支出，实际费用强制凭证，按币种汇总预算/实际/偏差；组合看板展示预算偏差，多币种不做错误合计。
BOM 物料成本有（成本 KPI），但**项目性支出**——模具费（动辄十几万）、认证费、NRE、打样费——只存在于任务 guide 文字里。对工厂这是最大的一笔项目现金流。建议：项目加简单的费用登记表（类别/预算/实际/凭证），组合看板加预算偏差列。不必做成财务系统，一张表就值回票价。

### 8. 纯软件 / OTA 发版无轨道 ✅ 已实施（2026-07-10）

> 落地：产品轴增加轻量软件发版单，保留基线、回归、灰度和回滚证据；创建人与 QA 验证人职责分离。安全相关、BOM 或制造影响更新被硬性拒绝并要求转 ECO。
固件修 bug 推 OTA 不动任何物料——走 ECO 太重（ECO 面向 BOM/产线），不走流程则无回归与回滚证据。建议：ECO 加"纯软件"裁剪档（跳过物料/产线任务，保留回归测试+版本记录+OTA 灰度回滚方案），或单独一张轻量「软件发版单」。

### 9. 平台 / 共用模块开发无归属
`platforms` 表还在（schema.ts:1478）但 PLM 简化后无流程消费。开发一颗给三个产品用的新机芯，项目挂在哪个产品下？现在只能挂其中一个，另两个产品的复用关系断链（where-used 只有 BOM 层）。建议：允许项目关联 platform 或多产品，暂不复活模块库。

### 10. 人员变动无交接流程 ✅ 已实施（2026-07-10）

移除成员前先盘点任务、待审批、交付物审核、条件项、行动项、费用、关闭移交责任和 PM 身份；必须选择在册接收人，所有转派与成员移除在同一事务完成，失败则不移除。
成员移除/离职后其 assignee 任务原地悬空，无"批量转派"操作，无离职检查清单（待审批交付物、评审人身份）。建议：成员移除时强制选择接收人，一键转派。

### 11. blocked_task 自动化规则已成死代码
状态语义修正（前置未完成=待开始）后，`blocked_task` 事件永远不会触发。要么删掉，要么将来接到"手动标记阻塞"功能上。属于本次修复的确定性后续，列入防遗忘。

## 🟢 低优先（记录在案，出现症状再动）

12. **跨项目问题复用检索**：同类失效在不同项目重复发生时无提示（issue 数据都在，缺"历史相似问题"检索/关联）。知识在库里，但不会自己走到人面前。
13. **SOP 的 SOP** ✅ 已实施（2026-07-10）：管理员发起受控变更，必须填写版本、适用轨道、变更原因、影响、迁移、回滚和生效日；只有指定的独立管理员可审批，申请人不能自批；批准后才能发布，所有状态变化写不可变审计事件。在途项目继续使用冻结的模板版本，不被新发布追溯改写。
14. **工期经验值无校准闭环**：schedule-graph 的 durationDays 是首版拍脑袋值，系统积累了大量"计划 vs 实际"数据却没有回填机制。跑完 3-5 个项目后做一次校准，预测才会越来越准。
15. **审核集中度**：deliverable 审核人默认落 PM，小团队里容易演变成 PM 一人盖章全项目。可加一个简单统计（单人审核占比）进管理层 KPI。
16. **委外开发轨道**：JDM/OBT 是"客户委托我们"，"我们委托别人"（买方案/外协设计）没有轨道——发生了就只能硬套 NPD。
17. **停产 EOL 流程** ✅ 已实施（2026-07-10）：产品轴增加停产计划、独立批准、七项证据清单和完成硬卡；完成后产品进入 `eol`，禁止继续发起 ECO 或轻量软件发版。
18. **新员工上岗**：模板库解决了"怎么填"，没有解决"怎么学"——建议留一个 demo 演练项目 + 各角色 30 分钟上手清单。

## 已核实为非盲点（扫描中排除）

- **客户数字化确认**：external_approval_instances 机制存在（OEM 变体链路），JDM Gate 签核以文件落地可审计——覆盖方式不同但闭环在。
- **样机去向**：F14 模板已含 SN/去向/异常登记。
- **审核绕行**：gate task 勾选封堵、自审自批禁止、评审鉴权等已在前几轮修复。
- **发布例外滥用**：conditional 强制发布已要求理由+负责人+截止并留痕（但其跟踪属盲点 4）。

## 建议的动手顺序（本轮已执行）

1（让步接收）与 4（条件项跟踪）最先做——都是"例外通道失守"类，机制小、防烂根；
2（证书台账+到期提醒）第二——合规硬伤，一条自动化规则+几个字段；
3+5（终止语义 + 关闭 Gate）第三——一起做，补全项目生命周期两端；
其余按 🟡🟢 顺序放入路线图。
