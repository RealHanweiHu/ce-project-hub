## 0. 现状盘点

| 类型 | 阶段 | 任务 | 交付物(去重) | 定位 |
|---|---|---|---|---|
| NPD | 7 | 55 | 40 | 0→1 全量 |
| ECO | 5 | 27 | 19 | 极小改（换料/降本/整改） |

| IDR | 4 | 26 | 22 | 外观翻新 |

**三个结构性问题：**

1. **DRV 交付物（51 项）比 NPD（40 项）还多**，且默认复用策略下 38/38 任务全保留——与「避免简单项目复杂化」的设计目标相反（详见 DRV 审核文档）。
2. **同名交付物跨阶段重复审核**：审核集按阶段隔离（`getReviewSatisfiedSet(projectId, phaseId, …)`），UN38.3/MSDS/电芯认证等证据在 DRV 的 P2/P4/P5 三处出现，同一份文件要审 2-3 次。
3. **任务粒度按"职能动作"切分而非"决策节点"切分**：如 ECO P1 用 7 个任务产出 4 份文档，其中 ep6 CCB 评审与 ep7 Kickoff 是背靠背的两次会议。

## 1. 删减原则（不可动 vs 可动）

**不可动（红线）：**
- 每类唯一的 MP Release 硬闸口阶段及其 Gate 任务（`assertNoReleaseGateTailoring` 已有守卫）；
- 电池安全/认证证据链：UN38.3、MSDS、电芯/电池包安全认证、EOL 100% 测试能力（保留"或复用确认"快速路径）；
- JDM/OBT 的客户签核交付物（规格确认书、外观签核、样机确认、DVT 确认、golden sample、放行）；
- 整机回归、P0/P1 关闭、Gate 评审任务本身。

**可动（四种手法，优先级从高到低）：**
- **合并**：背靠背的会议/文档合并为一个任务、一份交付物；
- **条件化**：任务只在特定产品属性/变更域下出现（机制批次二实现，短期由 PM 用现有任务级裁剪代替）；
- **降为指引**：不值得单独立任务跟踪的动作，降为上游任务 guide 里的一个步骤；
- **去重**：同名交付物只在一个 Gate 必交，其余阶段引用继承。

## 2. 各类型删减明细

### 2.1 NPD（55 → 50 任务；交付物 40 → 37）

| 动作 | 对象 | 处理 | 理由 |
|---|---|---|---|
| 合并 | c1 竞品分析 + c2 VoC | →「市场与用户洞察」 | 同一批人同一时段的输入侧工作；c2 guide 要求 15 深访+200 问卷，对 B2B 泵类工厂不现实，改为"客户/渠道反馈汇总"口径 |
| 合并 | c3 概念定义 + c5 商业可行性 | →「产品概念与商业测算」 | 概念书本就应含商业模型；交付物「商业可行性分析」并入「产品概念书」 |
| **保留** | p1 PRD 与 p2 PSD | **不合并**（2026-07-09 拍板） | **PSD 产品规格书要对外发给客户**，必须是独立、可单独发放的文档；PRD 是内部需求文档，两者受众不同 |
| 合并 | p3 时程规划 + p6 团队组建 | →「项目计划与资源（含 RACI）」 | 同为 PM 的一份计划的两个章节 |
| 合并 | mp2 良率监控 + mp3 产能爬坡 | →「量产爬坡与良率监控」 | 同一份周报的两列 |
| 删除 | mp4 工程变更管理 | 删 | 量产后变更走 ECO 项目闭环（系统已有 ECO 轨），项目内不再挂常设任务 |
| 条件化 | d5 SW架构、d3/d4 EE/PCB、d7a/d7b/p5a/d6a 电池族 | 批次二 | 纯机械产品（手打/自行车泵）无软件无 PCBA 无电池——见 §4 产品属性裁剪 |

保留不动：P4 EVT / P5 DVT / P6 PVT（发布闸口）全部任务；d6a/d7a/d7b 电池安全任务对含电池产品保留（2026-07 刚收紧，且带复用快速路径）。

### 2.2 ECO（27 → 18 任务；交付物 19 → 15）

ECO 现定位是"极小改"，但 P1 就有 7 个任务，比很多变更本身还重。

| 动作 | 对象 | 处理 | 理由 |
|---|---|---|---|
| 合并 | ep3 BOM差异 + ep2 影响范围评估 | →「影响分析（含 BOM 差异）」 | BOM 差异是影响分析的一节；交付物「BOM差异对比」并入「影响分析报告」 |
| 合并 | ep4 变更时程 + ep5 资源与供应商 | →「变更计划（时程+资源）」 | 极小改的时程和资源本就是一页纸 |
| 合并 | ep6 CCB评审 + ep7 ECO Kickoff | →「CCB 评审暨 Kickoff」（gateTask） | 背靠背两次会议合为一次；CCB 批准即启动 |
| 合并 | ed1 硬件 + ed2 结构 + ed3 软件变更 | →「变更设计实施」单任务 | 多数 ECO 只动一个域；三个域任务在单域变更时是两个空任务。变更域在 ECR 里勾选，guide 分域给操作步骤 |
| 合并 | ed4 DFM变更评审 + ed5 认证影响评估 | →「变更评审（DFM+认证影响）」 | 极小改场景一次会议覆盖 |
| 合并 | ev2 变更点专项验证 + ev3 回归测试 | →「变更验证与回归」 | 同一批样机同一轮测试 |
| 条件化 | ev4 可靠性关键项 | 批次二 | 只在变更触及结构/材料/电气应力时出现 |
| 合并 | em1 量产监控 + em2 效果验证 | →「变更后监控与效果验证」 | 同一监控窗口的两个结论 |

结果：P1 7→4，P2 6→3，P3 5→4（ev4 条件化前先保留），P4 5（🔒不动），P5 4→3。

## 4. 实施批次（本计划先冻结，动代码前逐批确认）

**批次一：纯模板合并（低风险，先做）**
改 `shared/sop-templates.ts` / `task-deliverables.ts` / `schedule-graph.ts` 的任务与交付物定义。在途项目兼容：删减是"放松"方向，不产生翻红——已建项目保留旧任务照常执行，新项目用新模板；可选提供一次性脚本把在途项目的被合并任务标 skipped（复用 gate-tightening manifest 的思路，方向相反、无需豁免审批）。

**批次二：产品属性裁剪（把 DRV 策略机制泛化）**
给项目加 3-4 个产品属性开关（含电池 / 含PCBA / 含软件 / 新开模具），NPD/ECO/IDR 的电池族、软件族、模具族任务按属性自动裁剪——机制完全复用 DRV 的 `getDerivativeEffectiveTaskIds` 模式（恒留集 + 属性→任务映射 + skipped 落库）。这是消化"纯机械产品被迫走全量 NPD"的正解。

**批次三：策略自动裁剪交付物与跨 Gate 继承（配合 DRV 修复一起做）**
- **策略/属性 → 自动删除相关任务、评审要求和交付物**（2026-07-09 拍板：不是"建议+PM确认"，而是应用策略时直接生效）：消费 `DERIVATIVE_REUSE_MODULE_RULES.deliverables` 映射自动落 override（理由自动填"模块直接复用—策略裁剪"），活动日志留痕、PM 可逐项恢复；四项安全/认证锚点（UN38.3/MSDS/电芯认证/EOL 100%）永不自动删除；
- 模块→任务/交付物映射本身按合理性逐条复核调整（如 dp6 冗余映射、direct_reuse 档位下哪些评审内容随之消失）；
- 同名交付物跨阶段审核继承（早阶段已审通过的证据在后续 Gate 自动满足，除非文件有新版本）；
- 修复 DRV 审核文档中的 P1 问题（排期依赖收缩、rebased 重置范围）。

**每批次验收口径**：`node scripts/test.mjs` 全绿 + 模板结构审计（阶段数/唯一发布闸口/gateTaskId 指向）+ 一个在途 demo 项目升级演练。

## 5. 交付物参照模板（配套）

删减解决"要交多少"，模板解决"怎么交"。全部 ~150 个交付物名称归并为 **12 个模板族**（评审记录 / 测试报告 / 问题清单 / 影响分析 / BOM及差异 / 项目计划 / 设计包清单 / 试产良率报告 / SOP·WI / 认证证据归档 / 复用确认单 / 客户签核单），每族一个带填写指引的标准模板。模板设计稿见本次输出的「交付物参照模板库」Artifact；落库方案（docs/templates/ + 交付物条目旁挂模板链接）在批次三之后实施。
