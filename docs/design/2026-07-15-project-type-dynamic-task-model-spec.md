# 项目类型与动态任务模型修改规格

> 日期：2026-07-15
>
> 状态：已确认，实施中
>
> 范围：NPD、DRV、ECO、JDM、OBT、产品库 Revision 的分流规则，以及 DRV/JDM/OBT 的动态任务生成
>
> 原则：项目创建保持轻量；已经立项的结论直接转成执行任务，不在项目内重复分析立项理由
> 数据前提：系统尚未正式投入使用，现有项目均为测试数据，不保留历史模板兼容

## 1. 目标

本次修改要解决四个问题：

1. 创建人可以直接选择项目类型，不再先选择“默认工作流”和“附加工作包”。
2. DRV 按创建时冻结的模块复用基线生成任务；JDM 在产品定义 Gate 冻结模块复用基线后生成后续任务，任务数量与模块选择真实联动。
3. JDM 与 OBT 使用不同的客户输入逻辑：JDM 由我方补全产品定义，OBT 以客户设计输入为基线。
4. 小改不进入项目；ECO 生成现有产品的受控版本，Revision 留在产品库，NPD/DRV/JDM/OBT 完成后生成独立产品。

预期结果：建项时只填写完成当前阶段所需的最少信息。DRV/OBT 创建时生成执行任务；JDM 创建时先生成产品定义阶段，定义 Gate 后生成后续任务。任何类型都不要求项目经理再到设置页二次裁剪。

## 2. 业务边界与分流

| 类型 | 何时使用 | 创建前提 | 任务来源 | 结果 |
|---|---|---|---|---|
| NPD | 全新产品、平台或核心架构 | 已确认为什么做、要做成什么样 | 固定全流程 | 独立产品 |
| DRV | 基于成熟产品模块开发衍生产品，本质是 NPD 减负 | 产品定义/规格基线已确认 | 公共任务 + 未复用模块任务包 | 独立产品 |
| ECO | 对现有产品做影响面较大的设计变更，如更换电机、PCBA | 现有产品及变更目标明确 | 公共变更任务 + 变更类型任务包 | 现有产品的受控版本 |
| JDM | 客户委托我方完成产品设计，客户通常只有 ID 图或概念 | 有客户概念输入和客户确认责任人，不要求客户已有产品规格书 | JDM 定义任务 + DRV 公共任务 + 未复用模块任务包 + 客户确认任务 | 独立客户产品 |
| OBT | 客户提供可制造的设计/BOM，我方负责导入、验证和量产 | 有可冻结的客户设计输入基线 | 输入/DFM、样件、试产、量产导入和客户放行任务 | 独立客户产品 |
| Revision | 包装、印刷、标签、文案等轻微变化 | 已有产品 | 产品库轻量受控变更 | 产品 Revision，不创建项目 |

补充分流规则：

- 供应商或二供变化不属于 DRV；按影响程度进入产品库受控变更或 ECO。
- 仅包装、标签、文案变化进入 Revision。
- 对运输、认证或量产放行有重大影响的包装变化进入 ECO。
- 新增目标市场本身不构成 DRV；按影响程度进入 Revision 或 ECO。
- OBT 如果需要我方新建设计图纸、BOM、软件或核心接口，应转为 JDM。
- DRV 六个模块全部复用时不能建 DRV，应返回重新判断 Revision、ECO 或无需项目。

## 3. 角色与批准规则

### 3.1 角色

- 产品经理/产品负责人：确认产品定义，创建或批准创建项目，批准所有设计范围变化。
- 项目经理：负责排期、任务跟进、跨部门协调、Gate 和交付闭环。
- 客户确认责任人：JDM/OBT 中负责取得并登记客户书面确认。
- 专业工程师：完成对应模块的设计、验证和证据交付。

系统字段建议：
- 新增项目级 `productOwnerUserId`，默认取当前创建人，管理层可在创建时改选。
- 保留现有 `pmUserId`，其业务含义明确为项目经理。
- 不直接用 `createdBy` 代替产品负责人，因为管理员可能代建项目。

### 3.2 各类型冻结时点

“创建即冻结”不能对所有项目类型一刀切：

| 类型 | 创建时冻结 | 后续正式冻结 | 冻结后的变化 |
|---|---|---|---|
| NPD | 已批准的产品定义、目标和完整流程 | 各阶段 Gate 继续冻结设计输出 | 产品负责人批准 |
| DRV | 产品规格基线、六模块复用状态、复用证据、风险声明 | 设计 Gate 冻结设计输出 | 产品负责人批准 |
| JDM | 只冻结客户原始输入快照、商务边界和客户确认责任人 | 产品定义 Gate 冻结我方产品规格书、CSR、六模块复用状态和风险声明 | 产品负责人批准后取得客户书面确认 |
| OBT | 客户设计输入版本和原始文件快照 | P1 Gate 冻结完成 DFM/缺口处理后的可制造输入基线 | 产品负责人批准后取得客户书面确认 |
| ECO | 现有产品基线、变更类型、变更目标和风险声明 | 设计/变更 Gate 冻结变更方案 | 产品负责人批准 |

JDM 在产品定义 Gate 前属于“定义收敛期”：模块复用判断、规格和 CSR 可以正常调整，不创建受控设计变更，也不要求客户对每次中间调整签字。Gate 通过时一次性确认正式基线；只有 Gate 之后的变化才进入受控变更。

第一增量尚未提供完整受控变更能力，因此所有已冻结基线在界面和接口上均为只读。变更入口置灰并提示“受控变更将在下一增量开放”，不得出现允许修改但不留审计的真空期。

已完成或待审批任务不得因后续模块状态变化被静默删除；第二增量开放变更时必须提示影响并保留审计记录。

## 4. 项目创建体验

创建流程保持两步：

1. 选择项目类型。
2. 填写项目基本信息和该类型必要的结构化信息，然后直接创建。

不再展示：

- 默认工作流程选择；
- 附加工作包选择；
- “是否真的需要建项目”提醒卡；
- 与产品属性重复的工作包选项。

### 4.1 通用字段

- 项目名称、项目编号、目标日期；
- 产品负责人（默认当前创建人）；
- 项目经理；
- 安全/法规结构化输入或“待评估”状态；
- 类型专属字段。

### 4.2 DRV 创建字段

- 产品定义/规格基线引用；
- 六个模块各选择“复用”或“不复用”；
- 每个复用模块填写来源产品/模块、型号或版本、证据引用和适用边界确认；
- 页面实时显示将生成的公共任务数、模块任务数和总任务数。

默认值全部为“不复用”，避免因为漏选而错误删减工作。任何轻微改动也选择“不复用”，由对应工程师完成跟进和确认。

### 4.3 JDM 创建字段

- 客户概念/ID 输入名称或引用；
- 客户料号（可选，客户未提供时不阻塞）；
- 客户特殊要求原始说明（可选，后续通过任务结构化）；
- 客户确认责任人；
- 商务边界。

JDM 创建时不要求 `customerInputVersion`、客户产品规格书或六模块复用状态。创建后只生成 P1 产品定义阶段；由我方编制产品规格书和客户特殊要求清单，并在该阶段登记六模块复用状态。产品定义 Gate 展示后续任务预览，客户确认并通过 Gate 后才原子生成 P2-P6 阶段和任务。

### 4.4 OBT 创建字段

- 客户设计输入版本；
- 客户料号；
- 客户确认责任人；
- 商务边界；
- “BOM 是否存在未指定模块”；如选择是，可勾选拟采用我方标准模块的范围。

OBT 创建后先做输入完整性和 DFM 检查。BOM 未指定的模块可以提出我方已有标准模块，但必须记录型号、接口、成本、交期、验证范围并取得客户确认。若需要重新设计，则停止 OBT 并转 JDM。

## 5. DRV 动态任务模型

### 5.1 六个产品模块

| 模块 | 典型内容 | “复用”的含义 |
|---|---|---|
| 电池/能源系统 | 电芯、电池包、BMS、保护、充电和热路径 | 型号、接口、参数和适用边界均不变 |
| 核心功能部件 | 电机、泵、风机、机芯、传动等 | 型号、性能边界和装配接口均不变 |
| 电子硬件 | PCBA、主控、驱动、电源、传感器 | 硬件版本、接口和关键器件均不变 |
| 软件/连接 | 固件、控制、APP、OTA、通讯、错误码 | 软件版本或受控分支、接口和功能边界均不变 |
| 产品结构/模具 | 壳体、堆叠、装配、密封、散热、模具 | 结构和模具不需要设计、修模或重新验证 |
| ID/CMF | 造型、颜色、材质、纹理、光泽、外观标准 | DRV 中只允许完全不变；单纯颜色/文字变化应走 Revision |

ID/CMF 与结构/模具必须保持两个独立模块和任务包，因为责任工程师不同。在 DRV/JDM 中将 ID/CMF 设为“不复用”时，结构/模具也必须为“不复用”；用户仍可分别查看两个专业的任务。

这是一条服务端领域不变量，不只是 UI 联动：

- 前端选中 ID/CMF“不复用”时自动同步结构/模具；
- 共享组合函数遇到非法组合直接返回领域错误；
- 创建接口和 JDM 产品定义 Gate 接口必须再次校验，拒绝 `id_cmf=not_reused + structure_mold=reused`；
- API 直调、克隆和转轨路径使用同一校验函数，不能绕过。

### 5.2 任务生成公式

`DRV 任务 = 公共任务 + 所有“不复用”模块的完整任务包`

- “复用”不生成独立任务，也不生成该模块的设计开发任务包。复用证据通过结构化模块基线完成：来源产品/模块、型号或版本、证据引用、适用边界和确认人；P1 的“产品定义/规格基线确认”公共任务负责检查六模块基线是否完整。
- “不复用”始终生成完整模块任务包，不再区分适配验证、轻改、重新开发。
- 公共任务不因模块复用而删除。
- 当前“模块复用与影响分析”任务删除，因为该结论在立项和创建项目时已经完成。

因此公式中的“复用模块”为零个额外任务，但不是零证据。前端任务预览和服务端任务组合器都只能按上述公式计算，不能另行暗加“复用确认任务”。

创建时直接组合最终阶段、任务、交付物和 Gate 必交物，不再“先种完整模板，再把任务标记 skipped”，也不再靠旧 `derivative-deliverable-tailoring` 生成自动豁免记录。某模块复用时，该模块专属任务和专属交付物从有效模板中同时不出现；安全/认证公共硬证据继续由公共 Gate 标准保留。

### 5.3 公共任务

公共任务至少包含下列独立任务，不能重新合并成一个综合验证任务：

1. 产品定义/规格基线确认；
2. 项目计划、RACI 和版本基线；
3. DFM、DFT、DFMEA 与验证计划；
4. 整机样机/Build 与版本记录；
5. 整机功能和兼容性回归；
6. 软件功能验证；
7. 可靠性验证（内部测试）；
8. 安规/认证验证；
9. 配件确认；
10. 包装验证；
11. 物流验证；
12. 治具确认；
13. EOL/测试程序确认；
14. DVT/验证问题关闭；
15. PVT 试产、良率和问题关闭；
16. 产品文件、BOM 和量产交付；
17. 发布、稳定期和项目关闭 Gate。

上述公共验证任务可以根据复用情况调整验证内容和证据深度，但任务本身保留，以确保有人确认集成结果。

### 5.4 模块任务包

| 模块 | 不复用时生成的核心任务 |
|---|---|
| 电池/能源系统 | 需求与设计、样件/集成、性能与保护验证、电池专项验证证据 |
| 核心功能部件 | 规格与选型/设计、样件、性能/温升/噪音/寿命专项验证 |
| 电子硬件 | 原理图/PCB/BOM 设计、硬件样件与联调、硬件专项验证 |
| 软件/连接 | 软件需求与设计、开发与联调、异常/接口/连接专项验证 |
| 产品结构/模具 | 结构设计、投模评审、模具开发、T1/T2 和结构专项验证 |
| ID/CMF | ID/CMF 设计、颜色/材质/工艺样件、外观标准与限度样本确认 |

“关键模块专项验证”必须拆入各模块任务包，不保留跨多个模块的模糊总任务。

### 5.5 DRV 阶段与 Gate 骨架

DRV 保留现有六阶段骨架和 phaseId，重写阶段内任务与 Gate 标准：

| 阶段 | 主要任务 | Gate 与必备证据 |
|---|---|---|
| `iteration` / P1 迭代启动 | 产品定义/规格基线确认、结构化复用证据检查、项目计划/RACI、风险声明确认 | 迭代 Kickoff；规格基线、六模块状态与复用证据、风险结论、计划和责任人完整 |
| `design` / P2 设计 | 所有不复用模块的设计任务、DFM/DFT/DFMEA、验证计划、配件定义 | 设计冻结；模块设计输出、BOM/接口、验证矩阵和产品负责人批准完整 |
| `evt` / P3 EVT | EVT Build、各不复用模块专项验证、整机功能/兼容回归、软件功能验证、问题关闭 | EVT Gate；Build 版本、模块验证、整机与软件结果、P0/P1 问题处置完整 |
| `dvt` / P4 DVT | DVT Build、可靠性内部测试、安规/认证验证、配件确认、包装验证、物流验证、问题关闭 | DVT Gate；可靠性、安全/认证、包装物流、配件和问题关闭证据完整 |
| `pvt` / P5 PVT | 治具确认、EOL/测试程序确认、试产、良率/产能、文件/BOM 发布、量产移交 | 量产发布 Gate；治具和 EOL 分别验收、试产达标、文件齐套、发布硬证据完整 |
| `mp` / P6 稳定与关闭 | 稳定期数据、遗留问题关闭、正式移交、项目关闭 | Close Gate；继续保留 `isCloseGate=true` 及现有发布后稳定、条件项和移交硬卡 |

各阶段继续显式维护 `gateStandard` 的 entryCriteria、exitCriteria、requiredDeliverables、responsibleRoles、evidenceRequirements 和 exceptionStrategy。任务、`TASK_DELIVERABLES`、Gate 必交物和排期依赖必须使用同一组新任务 ID 一起修改，不能只替换模板任务数组。

## 6. JDM 两段式任务模型

JDM 使用 DRV 的六个模块和二元复用规则，但模块状态不能在创建时冻结。系统采用两段式生成：

1. 创建项目时只插入 `input` 阶段及其任务，保存客户原始输入快照，不插入 P2-P6 阶段和任务。
2. 产品定义 Gate 通过时，在同一事务中冻结产品规格书、CSR、模块复用基线和风险声明，并按确认后的模块状态生成 P2-P6 阶段和任务。

Gate 通过前，模块状态是可编辑的定义草稿；Gate 通过后，模块状态成为只读执行基线。前端 Gate 预览与服务端实际生成必须调用同一纯函数，确保任务数一致。

### 6.1 设计前强制任务

1. 客户概念/ID 图和原始要求接收；
2. 需求澄清与缺口清单；
3. 我方编制产品规格书；
4. 客户特殊要求清单（CSR）；
5. 设计责任、接口和商务边界确认；
6. 模块复用草稿登记，并为复用模块补齐来源、版本、证据和适用边界；
7. 产品负责人内部确认；
8. 客户确认产品规格书和 CSR；
9. 产品定义 Gate：冻结正式基线并生成后续任务，未确认不得进入设计。

CSR 至少能记录：外观缝隙/段差、颜色/光泽/纹理、外观缺陷标准、按键手感、噪音、振动、温升、Logo、配件、包装、客户测试方法、目标值、判定方法、限度样本、责任人和确认记录。

### 6.2 设计与验证

产品定义 Gate 通过后的完整公式：

`JDM 任务 = JDM 定义任务 + DRV 剩余公共任务 + 所有“不复用”模块任务包 + 客户确认任务`

JDM 定义阶段中的“产品规格与执行基线确认”已经承担 DRV 公共任务 1“产品定义/规格基线确认”，组合器必须按语义键去重，不能在 P2 再生成一项同义任务。

客户确认任务至少覆盖：

- 产品规格书和 CSR；
- ID/CMF；
- 设计冻结；
- EVT 样件/结果；
- DVT 样件/结果；
- 配件和包装；
- Golden Sample；
- 量产放行。

冻结后任何设计修改都要先由产品负责人批准，再取得客户书面确认并形成新基线。

### 6.3 JDM 阶段与 Gate 骨架

JDM 保留现有六阶段 phaseId；P1 先创建，P2-P6 在 P1 Gate 通过后生成：

| 阶段 | 主要任务 | Gate 与必备证据 |
|---|---|---|
| `input` / P1 产品定义 | 客户概念接收、需求缺口、我方产品规格书、CSR、责任边界、模块复用草稿、风险声明、内审和客户确认 | 产品定义冻结 Gate；客户确认的规格/CSR、模块正式基线、风险结论和责任边界完整；批准动作原子生成 P2-P6 |
| `design` / P2 详细设计 | 不复用模块设计、DFM/DFT/DFMEA、验证计划、ID/CMF/结构接口、客户设计确认 | 设计冻结 Gate；产品负责人批准且客户确认设计基线 |
| `evt` / P3 EVT | EVT Build、模块专项验证、整机/软件功能验证、问题关闭、客户样机确认 | EVT Gate；内部验证结论和客户 EVT 确认完整 |
| `dvt` / P4 DVT | 可靠性、安规/认证、配件、包装、物流、DVT 问题关闭、客户 DVT 确认 | DVT Gate；独立验证证据与客户确认完整 |
| `pvt` / P5 PVT | 治具、EOL/测试程序、试产、良率/产能、Golden Sample、客户量产放行 | MP Release Gate；客户签样和量产放行、内部发布硬证据完整 |
| `mp` / P6 稳定与客户移交 | 稳定期、遗留问题、客户/产品维护移交、关闭 | Close Gate；保留 `isCloseGate=true` 和现有关闭硬卡 |

JDM 的 gateStandard 在 DRV 对应 Gate 基础上增加客户会签槽位和客户证据，但不能用客户签字替代内部可靠性、安全或量产放行证据。

## 7. OBT 任务模型

OBT 不使用 DRV 的“复用/不复用”来裁剪客户设计。它围绕客户设计输入的完整性、可制造性和量产导入生成任务。

### 7.1 固定任务组

1. 客户设计输入接收与版本冻结；
2. BOM、图纸、规格、软件和测试输入完整性检查；
3. 设计/BOM 缺口清单；
4. DFM、DFA、DFT 和制造风险评审；
5. 我方标准模块建议（仅客户未指定时）；
6. 客户确认缺口处理和标准模块建议；
7. 物料、供应、样件和首件准备；
8. 治具设计/确认；
9. EOL/测试程序设计与验收；
10. 首件、功能、可靠性和客户样件确认；
11. 包装验证；
12. 物流验证；
13. PVT、良率、产能和问题关闭；
14. Golden Sample/客户放行；
15. 量产移交和关闭。

### 7.2 转轨与变更

- 输入冻结后任何修改都要记录原因、影响、客户书面确认和新基线。
- 如果缺口需要我方完成新的产品设计，不得继续用 OBT 缩短流程，应受控转为 JDM。
- 治具与 EOL/测试程序保持两个独立任务。

### 7.3 OBT 阶段与 Gate 骨架

OBT 保留现有四阶段 phaseId，重写任务和 Gate 标准：

| 阶段 | 主要任务 | Gate 与必备证据 |
|---|---|---|
| `intake` / P1 设计接收 | 客户输入快照、完整性检查、BOM/设计缺口、DFM/DFA/DFT、风险声明、标准模块建议、客户确认 | 转产受理与可制造输入冻结 Gate；客户确认的输入版本、缺口处置、标准模块建议、风险与责任边界完整 |
| `sample` / P2 打样与首件 | 物料/样件准备、治具、EOL/测试程序、FAI、功能/可靠性验证、客户样件确认 | FAI 与客户签样 Gate；治具和 EOL 分别验收，首件与验证结果合格，客户签样完整 |
| `pvt` / P3 试产 | 包装验证、物流验证、试产、良率/产能、问题关闭、文件齐套、Golden Sample | MP Release 与客户放行 Gate；试产指标、包装物流、Golden Sample、客户放行和内部发布硬证据完整 |
| `mp` / P4 稳定与客户移交 | 稳定期、遗留问题、客户/产品维护移交、关闭 | Close Gate；保留 `isCloseGate=true` 和现有关闭硬卡 |

OBT 的 `gateStandard` 必须显式包含客户会签，但客户确认不能替代我方制造、质量、安全或发布责任。

## 8. ECO 与 Revision

### 8.1 ECO

ECO 面向现有产品的大影响设计变更，输出现有产品的受控版本，不生成独立产品。创建时直接选择变更类型，系统生成对应任务包，例如：

- 电池/能源系统变更；
- 核心功能部件变更；
- 电子硬件变更；
- 软件/连接变更；
- 结构/模具变更；
- 制造、治具或 EOL 变更；
- 重大包装运输或认证变更。

每个任务包至少包含变更设计、影响确认、专项验证、回归、文件/BOM 更新和版本切换。所有设计变更均由产品负责人批准。

### 8.2 Revision

Revision 只在产品库维护，适用于包装、印刷、标签、说明书和文案等轻微改版。Revision 不依赖项目，也不生成项目任务；若影响运输验证、认证、BOM 核心设计或多人复杂协作，则升级为 ECO。

### 8.3 安全/法规声明与认证硬卡

六模块复用状态不能取代现有 `changeScopeDeclaration → deriveSopRiskAssessment → certificateCoverage → Gate` 链路。两者职责不同：模块状态决定专业任务包，风险声明决定安全/法规等级、会签和认证证据。

保留规则：

- `projectChangeScopeDeclarations` 的版本记录、风险单向升级、证书覆盖计算和发布/关闭硬卡全部保留。
- 不通过备注、规格书或任务描述关键词推断风险，仍只使用结构化字段。
- 模块选择用于减少重复输入和控制显示，但服务端最终仍生成并保存完整 `ProjectChangeScopeDeclaration`。

各类型采集方式：

| 类型 | 风险声明时点与方式 |
|---|---|
| NPD | 创建时完成完整结构化声明，固定全流程不等于可以跳过风险和认证输入 |
| DRV | 创建时完成；复用模块自动锁定其“不发生设计变化”的相关项，不复用模块只展示相关追问；目标市场和认证覆盖仍独立确认 |
| JDM | 创建时可为“待评估”，不得保存一份全 false 的假声明；P1 产品定义 Gate 前必须基于已确认规格/CSR 完成声明和认证路径 |
| OBT | P1 Gate 前基于客户设计输入完成；客户输入版本冻结不代表安全/法规已覆盖 |
| ECO | 创建时保留完整变更范围声明，并同时驱动 ECO 变更任务包和风险/认证硬卡 |

示例：电池模块“复用”可以自动确定没有新电芯、电池包、BMS 或保护参数设计变化，但仍需确认复用证据的适用市场和有效性；电池模块“不复用”不能自动等同所有电池风险项都为 true，必须回答与电芯、电池包/BMS、保护参数和热边界相关的最少追问。

JDM 的“待评估”状态由“尚无风险声明版本”表示，产品页面明确显示待评估，P1 Gate readiness 直接阻塞；不新增一个可被误当成已评估结果的默认低风险声明。

### 8.4 NPD 单一全流程决策

本规格确认 NPD 只有一个完整流程，创建人不选择 lite/standard/full，也不选择 battery/cert/software/mold 附加包。此前“NPD 分档 + 附加包”的设计在与本规格冲突的部分被废止；后续可以精简单一全流程内部的任务内容，但不能重新给建项者增加档位或工作包选择。

实施时删除而不是闲置以下死逻辑：

- `NpdTemplateTier`、档位推荐、降档判断和 lite 阶段映射；
- 用户可选 `NpdAddonPackId` 和附加包组合；
- `npdTemplate`、`npdTemplateDowngradeReason` 创建输入与项目自定义字段审计；
- `NPD_FULL_TEMPLATE_CONFIG` 在前后端之间重复传递的固定值。

保留一套固定 NPD 阶段、完整专业任务线、红线任务、结构化风险声明和认证硬卡。相关旧规格文档必须标注“已由 2026-07-15 项目类型规格取代”，避免两份生效规格继续冲突。

### 9.2 结构化配置

项目在 `customFields` 保存创建基线：

```ts
type ModuleReuseState = "reused" | "not_reused";

interface ModuleReuseEvidence {
  sourceRef: string;
  modelOrVersion: string;
  evidenceRef: string;
  boundaryConfirmed: boolean;
}

interface ProjectExecutionBaseline {
  modelVersion: "project-track-v1";
  status: "draft" | "frozen";
  productDefinitionRef?: string;
  moduleReuse?: Record<ProductModuleId, ModuleReuseState>;
  reuseEvidence?: Partial<Record<ProductModuleId, ModuleReuseEvidence>>;
  customerConceptRef?: string;
  customerInputVersion?: string;
  proposedStandardModules?: ProductModuleId[];
  frozenAt?: string;
  frozenBy?: number;
}
```

DRV 创建时保存 `status=frozen`；JDM 创建时只保存客户原始输入快照和 `status=draft`，产品定义 Gate 通过时才补齐正式基线并改为 `frozen`；OBT 同时保存不可覆盖的客户原始输入快照和经过缺口处理后的执行基线。

正式使用后，变更不能覆盖原快照，应增加新版本或变更记录。第一增量可以直接调整现有测试数据结构；优先复用 `customFields` 和活动日志，只有在无法满足批准、查询或审计时才增加数据库表。

### 9.3 稳定任务 ID

- 当前模板可直接使用语义清晰的新任务 ID，不保留 `di*`、`dd*`、`de*` 等旧 ID 的兼容映射。
- 公共任务和模块任务包分别使用稳定前缀，例如 `drv_common_*`、`drv_battery_*`。
- JDM 复用同一模块任务定义时，通过模板构建器复制任务结构，而不是复制粘贴两套文本。

## 13. 测试策略

### 13.1 纯逻辑测试

- 六模块全部“不复用”时得到公共任务 + 六个完整任务包。
- 任一模块从“不复用”改为“复用”时，只减少该模块任务包。
- 复用模块不增加额外任务，但缺少来源、版本、证据或边界确认时基线校验失败。
- ID/CMF 不复用时自动保留结构/模具任务包；非法的“ID/CMF 不复用 + 结构复用”组合被共享校验函数拒绝。
- 六模块全部复用时 DRV 创建被拒绝。
- JDM = 定义任务 + DRV 组合任务 + 客户确认任务。
- JDM draft 基线只返回 P1；frozen 基线返回 P1-P6 和按模块组合的任务。
- OBT 标准模块建议不触发 DRV 模块裁剪逻辑。
- ECO 变更类型只生成对应任务包和公共变更任务。
- 每个新阶段都有 gateTaskId、gateStandard 和任务依赖闭包，不存在指向已删除旧 ID 的引用。

### 13.2 服务端测试

- DRV/OBT 创建后数据库任务数与前端预览一致。
- JDM 没有客户产品规格书也可以创建，创建后数据库只有 `input` 阶段和 P1 任务。
- JDM 产品定义 Gate 未确认前不能进入设计；Gate 通过时原子生成 P2-P6，任务数与 Gate 预览一致，重复提交不会重复插入。
- JDM P1 Gate 缺规格、CSR、模块正式基线、客户确认或风险声明任一项时均被阻塞。
- OBT 缺少客户输入版本时不能创建。
- API 直接提交“ID/CMF 不复用 + 结构复用”时被服务端拒绝。
- 第一增量中已冻结的模块/输入基线不能通过通用编辑、克隆或设置接口修改。
- 风险声明版本、风险升级、证书覆盖和发布/关闭硬卡在新模板下继续工作；JDM 没有声明时显示待评估并阻塞 P1 Gate。
- NPD 创建 API 不再接收档位、附加包或降档理由，固定生成唯一全流程。
- 清理测试项目后，新建各类型项目的阶段、任务、排期和 Gate 均能正常工作。

### 13.3 界面验收

- 项目类型可直接选择；不存在默认工作流和附加工作包。
- DRV 创建页模块只有“复用/不复用”，选择变化后任务数量和清单立即更新。
- JDM 创建页不要求模块状态；P1 产品定义页使用“复用/不复用”，并实时预览 Gate 后将生成的任务。
- DRV 创建后、JDM P1 Gate 通过后均无需再去项目设置应用一次策略。
- 第一增量中已冻结基线只读，变更入口置灰并明确提示后续开放受控变更。
- JDM 文案明确“我方编制规格并与客户确认”；OBT 文案明确“客户设计输入冻结”。

### 第二增量：受控变更

- Revision 保持产品库轻量入口。

## 15. 成功标准

- 建项者不需要理解模板、工作包或后续裁剪设置。
- DRV 创建预览、JDM 产品定义 Gate 预览均等于实际生成任务数。
- 复用一个模块只会减少该模块的设计与专项验证任务，不会删除公共集成、可靠性、安规、软件、包装、物流、配件、治具和量产确认责任。
- JDM 能在客户只有概念或 ID 图时合法启动，并在设计前完成我方规格和特殊要求的客户确认。
- OBT 能处理 BOM 未指定模块的情况，但未经客户确认不会采用我方方案，也不会把新设计伪装成 OBT。
- 所有设计范围变化均可追溯到产品负责人批准；JDM/OBT 还能追溯客户确认。
- 清理测试数据后，新模型成为唯一正式业务模型，不存在历史兼容分支。
