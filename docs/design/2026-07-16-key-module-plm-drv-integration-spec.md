# 关键模块 PLM 与 DRV 集成设计规格

> 状态：已确认，可进入实施拆解
> 日期：2026-07-16
> 决策人输入：产品/项目管理实际流程访谈
> 关联概念：`docs/ideas/key-module-digital-thread.md`

## 0. 文档效力与基线说明

本规格修订并覆盖以下旧设计中与“模块复用、项目发布、Product Revision”冲突的内容：

- `docs/design/2026-06-13-two-axis-plm-architecture.md`
- `docs/design/2026-06-13-project-axis-modular-sop.md`

冲突时以本规格和 `2026-07-15-project-type-dynamic-task-model-spec.md` 为准。特别是：

- IDR 不再是项目类型；复杂外观翻新归入 DRV。
- DRV 模块状态不是五级变更，而是“复用 / 不复用”二元状态。
- 项目创建、任务和 Gate 不依赖 Product 或 Product Revision。
- 项目完成后发布的是产品技术基线；Product Revision 只表示包装、印刷、标签、文案等轻微改版。
- 当前 `main` 尚未包含动态项目任务模型完整实现。实施必须先落地或重放 `codex/project-type-dynamic-task-model` 工作线，不能在两套项目创建逻辑上并行开发。

## 1. 目标

### 1.1 产品目标

1. 降低 DRV 立项填写负担，让用户通过选择受控模块完成复用声明。
2. 让模块复用真正减少对应设计开发任务，同时保留公共验证任务。
3. 用同一份受控模块引用同时驱动项目模块基线与工作 BOM，消除双录入和不一致。
4. 让产品库具备可用的关键模块资产、BOM 配置和技术基线，而不是整机与零部件混合的卡片集合。
5. 为后续 BOM 差异影响分析保留结构化基础，但不在 MVP 自动重写任务。

### 1.2 成功指标

- DRV 创建时除基本项目字段外，不再要求填写自由文本模块来源、型号和证据。
- 选择来源产品后，三个物理关键模块可在一次交互中自动带出；无来源产品也能直接搜索。
- 100% 的已批准关键模块都有唯一编号、固定内部 BOM、技术确认人和使用批准人。
- DRV 创建后，项目模块基线与工作 BOM 中的关键模块引用一致率为 100%。
- 所有“六模块零复用”的 DRV 请求在前端提交阶段被引导，在服务端被拒绝。
- 批量 BOM 导入在提交前能明确展示成功行、警告行和错误行；错误不得造成部分写入。

## 2. 角色与责任

| 角色 | 责任 |
|---|---|
| 产品经理/项目创建人 | 选择项目类型，确认 DRV 复用基线；任何设计变更需要产品经理同意 |
| 项目经理 | 推进项目；可批准模块用于项目；处理模块基线与项目范围的受控调整 |
| 工程师 | 创建关键模块、维护草稿内部 BOM、确认技术资料完整性 |
| 产品经理或项目经理 | 在技术确认后批准模块“允许项目使用”；两者任一具备批准权 |
| SCM | 维护供应商、二供、价格等商业信息；不通过修改模块编号表达二供变化 |
| QA/认证 | 在项目任务中执行可靠性、安规和认证验证；不承担模块主数据批准 |
| 系统管理员 | 配置模块编号规则、参数模板、品类词表和导入模板 |

批准是两个不同责任层：工程师对技术完整性负责，产品经理/项目经理对项目使用风险负责。创建者不能用单一动作替代两层记录。

## 3. 术语

| 术语 | 定义 |
|---|---|
| 模块类型 | 电池/能源、核心功能、电子硬件三类受控分类 |
| 关键模块 | 有独立模块编号、固定内部 BOM、技术确认和使用批准的可复用装配体 |
| 模块内部 BOM | 构成关键模块的全部受控子件及数量、规格和位号 |
| 产品 BOM | 构成产品技术基线的物料/模块清单 |
| 项目工作 BOM | 项目执行期间可维护、最终用于发布技术基线的 BOM |
| 项目模块基线 | 项目创建时锁定的六模块复用状态，以及三个物理模块的精确引用和快照 |
| 产品技术基线 | 项目发布后形成的不可变技术配置快照，包含关键模块和冻结 BOM |
| Product Revision | 产品完成后的轻微包装、印刷、标签、文案改版；不表示技术模块或设计版本 |
| 来源产品 | DRV 创建时可选的检索捷径，用来预填其当前技术基线的关键模块，不是立项前置条件 |

## 4. 范围

### 4.1 第一阶段受控关键模块

| moduleType | 中文名称 | 典型对象 |
|---|---|---|
| `battery_energy` | 电池/能源模块 | 电池包、电芯组合、BMS/保护板集成包 |
| `core_function` | 核心功能模块 | 电机、泵体、机芯、风机或机械执行组件 |
| `electronics_hardware` | 电子硬件模块 | PCBA、主控板、功率板或组合板卡 |

只有这三类进入关键模块库。普通电阻、线材、紧固件、包装物等仍是普通 BOM 行。

### 4.2 DRV 六模块

DRV 任务裁剪继续使用以下六类：

1. `battery`
2. `core_function`
3. `electronics`
4. `software_connectivity`
5. `structure_mold`
6. `id_cmf`

每类只允许 `reused` 或 `not_reused`。前三类与关键模块主数据映射；后三类第一阶段只保存状态。

### 4.3 非范围

- 供应商、二供、包装/标签/文案、重大包装运输或认证变化不作为 DRV 模块。
- 不做历史数据迁移。
- 不做多层递归 BOM 展开和自动跨产品 ECO 传播。
- 不在模块基线变化时自动恢复/删除项目任务。

## 5. 核心业务不变量

### 5.1 同一模块判定

1. 关键模块的身份由 `moduleNumber + 已批准内部 BOM` 决定，不由名称决定。
2. 只有全部受控子件、数量、规格和接口定义均未变化，才可视为同一模块复用。
3. 任一受控子件发生变化，必须创建新模块编号；不得原位修改已批准模块。
4. 新模块可从旧模块派生，必须保存 `derivedFromModuleId` 和可计算的 BOM 差异。
5. 已批准的同一子件增加合格二供，不改变模块编号。供应商资格与切换记录属于独立供应关系。

### 5.2 DRV 合法性

1. DRV 六模块中至少一个为 `reused`。
2. 若六模块全部为 `not_reused`，服务端拒绝创建，前端仅允许返回修改或切换为 NPD。
3. `id_cmf = not_reused` 时，`structure_mold` 必须为 `not_reused`；前端联动，服务端拒绝非法组合。
4. 三个物理模块选择 `reused` 时，必须引用状态为 `approved` 且未受限/停用的精确关键模块。
5. 三个物理模块选择 `not_reused` 时，不允许携带模块引用。
6. 来源产品始终可空，且不能作为 DRV 合法性的替代条件。

### 5.3 基线与 BOM 一致性

1. 创建 DRV 时，项目模块基线和项目工作 BOM 必须在同一数据库事务中提交。
2. 被复用的物理关键模块必须在工作 BOM 中有对应的受控引用。
3. 同一项目、同一物理模块类型只能有一个创建时基线引用。
4. 项目基线保存模块编号、模块 ID、内部 BOM 摘要和关键参数快照。来源产品后续变化不得影响已创建项目。
5. 基线创建失败或 BOM 写入失败时，项目创建整体回滚。

### 5.4 发布语义

1. 项目不需要已有 Product 或 Revision 才能创建。
2. 项目完成后可以创建独立 Product，或在明确关联时更新既有 Product 的技术基线。
3. 技术模块、设计 BOM 和规格的发布进入 `product_technical_baselines`，不生成 Product Revision。
4. Product Revision 只处理轻微包装、印刷、标签、文案变化，不进入项目流程。

## 6. 数据模型

表名为建议名；实现时遵循现有 Drizzle 和主键约定。

### 6.1 `key_modules`

| 字段 | 类型/约束 | 说明 |
|---|---|---|
| `id` | varchar(32), PK | 稳定内部标识 |
| `moduleNumber` | varchar, unique, not null | 受控模块编号 |
| `moduleType` | enum, not null | 三类模块之一 |
| `name` | varchar, not null | 可读名称 |
| `category` | varchar, not null | 产品/模块适用品类；允许跨品类搜索 |
| `model` | varchar, nullable | 对外型号或工程型号 |
| `attributes` | jsonb | 按模块类型模板保存关键参数 |
| `status` | enum | `draft / technical_confirmed / approved / restricted / obsolete` |
| `derivedFromModuleId` | FK nullable | 派生来源 |
| `evidenceRefs` | jsonb | 规格、图纸、测试等引用，不保存创建表单自由文本证据 |
| `createdBy` | FK | 创建工程师 |
| `technicalConfirmedBy/At` | FK/timestamp nullable | 技术确认 |
| `approvedBy/At` | FK/timestamp nullable | 项目使用批准 |
| `restrictionReason` | text nullable | 限制使用原因 |
| `createdAt/updatedAt` | timestamp | 审计字段 |

规则：`moduleNumber` 全局唯一；批准后所有定义字段与内部 BOM 只读。限制和停用是状态变化，不改历史快照。

### 6.2 `key_module_items`

| 字段 | 类型/约束 | 说明 |
|---|---|---|
| `id` | serial, PK | 子记录 |
| `moduleId` | FK, not null | 所属关键模块 |
| `partNumber` | varchar, not null | 内部料号 |
| `name` | varchar, not null | 名称 |
| `spec` | text | 规格/接口摘要 |
| `quantity` | decimal, not null | 用量 |
| `refDesignator` | varchar nullable | 位号/位置 |
| `componentProductId` | FK nullable | 已有受控零部件引用，可空 |
| `sortOrder` | int | 稳定展示顺序 |

供应商和二供不写入模块身份字段。商业信息继续受现有 BOM 权限保护。

### 6.3 `product_technical_baselines`

| 字段 | 类型/约束 | 说明 |
|---|---|---|
| `id` | varchar(32), PK | 技术基线 ID |
| `productId` | FK, not null | 所属产品 |
| `baselineLabel` | varchar, not null | 展示编号，格式可配置 |
| `sourceProjectId` | FK, unique, not null | 来源项目 |
| `keyModulesSnapshot` | jsonb | 三个关键模块引用与快照 |
| `bomSnapshot` | jsonb | 发布 BOM 快照 |
| `specSnapshot` | jsonb | 产品规格/接口快照 |
| `releasedBy/At` | FK/timestamp | 发布记录 |

产品保存 `currentTechnicalBaselineId` 指向当前技术基线。旧基线永久可追溯。

### 6.4 `product_module_assignments`

用于可查询地表达技术基线使用的关键模块，避免所有查询依赖 JSON：

| 字段 | 说明 |
|---|---|
| `technicalBaselineId` | 技术基线 |
| `moduleType` | 三类模块 |
| `moduleId` | 精确关键模块 |
| `moduleSnapshot` | 发布时快照 |

唯一约束：`technicalBaselineId + moduleType`。

### 6.5 `project_module_baselines`

| 字段 | 说明 |
|---|---|
| `projectId` | 项目 |
| `drvModuleKey` | 六模块之一 |
| `reuseState` | `reused / not_reused` |
| `keyModuleId` | 仅前三类复用时必填 |
| `sourceProductId` | 可空，仅记录选择来源 |
| `sourceTechnicalBaselineId` | 可空，记录自动带出来源 |
| `moduleSnapshot` | 精确编号、参数、内部 BOM 摘要 |
| `confirmedBy/At` | 创建时确认记录 |

唯一约束：`projectId + drvModuleKey`。

### 6.6 现有表调整

- `products`：增加 `currentTechnicalBaselineId`；保留 Product Revision，但收窄其语义。
- `bom_items`：增加 `keyModuleId` 与必要的快照/来源字段；工作 BOM 可同时包含普通物料和关键模块行。
- `project_revisions` 或现有执行基线相关表：不重复存储模块事实，只引用 `project_module_baselines`。
- Activity Log：记录模块创建、技术确认、批准、限制、派生、DRV 自动带出、用户覆盖、BOM 同步和发布。

## 7. 状态机与权限

```text
draft
  └─ 工程师技术确认 → technical_confirmed
       └─ 产品经理/项目经理批准 → approved
            ├─ 限制使用 → restricted
            └─ 停用 → obsolete

draft / technical_confirmed 可退回 draft
approved 不可退回编辑；需要修改时“基于此模块新建”
```

- 技术确认前必须有模块编号、模块类型、名称、品类、必填关键参数和至少一条内部 BOM。
- 使用批准前必须存在技术确认人和时间。
- 创建者如果同时具备项目经理权限，仍需形成两个不同动作；是否允许同一自然人完成两步由权限配置决定，默认允许但完整留痕。
- `restricted`/`obsolete` 模块不得用于新项目；已有项目和技术基线仍显示历史记录与状态。

## 8. DRV 创建交互

### 8.1 页面内容

只显示：

- 项目名称、项目编号、产品品类、项目经理、产品负责人、计划日期等基本字段；
- 可选来源产品；
- 六模块“复用 / 不复用”选择；
- 三个物理模块在复用时显示已选受控模块。

不显示：项目目标、规格书引用、安全/法规影响确认、自由文本复用证据和默认工作流/附加工作包。产品规格书在项目任务中提交。

### 8.2 来源产品预填

1. 用户搜索并选择来源产品。
2. 系统读取其当前产品技术基线。
3. 对三个物理模块：有已批准引用则默认 `reused` 并带出；没有则保持 `not_reused`。
4. 用户可覆盖任何结果；覆盖行为进入审计记录。
5. 清除来源产品不清空用户已确认的模块选择，但界面标记其已变为直接选择。

### 8.3 直接模块搜索

- 支持模块编号、名称、型号、关键参数和品类关键词。
- 同品类优先，非同品类仍可见；明确显示品类标签。
- 默认只返回 `approved`；受限和停用模块只在历史查看中出现。
- 结果显示模块编号、名称、模块类型、型号、关键参数摘要、批准状态和 where-used 数量。
- 使用可搜索组合框，支持键盘导航、清除、空状态和加载状态。

### 8.4 零复用提交弹窗

只有用户点击“创建项目”且六模块全部为 `not_reused` 时显示：

- 标题：`当前没有复用任何模块`
- 内容：`DRV 至少需要复用一个现有模块；如果全部模块重新开发，更符合 NPD。`
- 操作：`返回修改`、`切换为 NPD`

没有“仍按 DRV 创建”。切换为 NPD 时保留项目名称、编号、品类、产品负责人、项目经理和日期，丢弃/忽略所有 DRV 复用字段。

## 9. 任务裁剪

任务公式保持：

```text
DRV 任务 = 公共任务 + 所有 not_reused 模块的专项任务包
```

- `reused` 模块不生成该模块的设计开发任务包。
- 复用判断已经在初步立项阶段完成，创建页不再生成“模块复用和影响分析”任务。
- 公共任务必须保留产品规格确认、计划启动、整机集成、可靠性（内部测试）、安规、认证、软件功能验证、包装、物流、配件确认、治具、试产和结项等已确认任务。
- 关键模块专项验证分别存在于相应模块任务包，不合并成一个笼统任务。
- JDM/OBT 可复用同一关键模块选择器，但其阶段、客户确认和任务模板继续按各自项目类型执行。

## 10. 原子创建与 BOM 同步

服务端创建 DRV 的顺序：

1. 校验项目基本字段和六模块不变量。
2. 锁定/读取所选关键模块，确认仍为 `approved`。
3. 创建 Project。
4. 写入六条 `project_module_baselines` 和物理模块快照。
5. 将三个被复用的物理关键模块写入项目工作 BOM；同类型已有自动行时幂等更新。
6. 根据六模块状态组合并种入任务。
7. 写 Activity Log。
8. 提交事务。

任何一步失败全部回滚。客户端不得分别调用“创建项目”和“补写 BOM”来模拟原子性。

## 11. BOM 批量导入

### 11.1 流程

```text
上传 Excel/CSV
  → 选择工作表
  → 字段映射
  → 预览与规范化
  → 匹配现有普通物料/关键模块
  → 错误与重复处理
  → 原子提交到工作 BOM
```

### 11.2 校验

- 必填：名称、数量；料号按模板要求。
- 数量必须为正数；同一料号/位号重复需合并或明确报错。
- 模块编号匹配关键模块库时，必须校验类型与状态。
- 未匹配但疑似关键模块的行只能创建 `draft` 候选，不能直接写成已批准模块。
- 普通物料永远不会因名称含“电池/电机/PCBA”自动进入关键模块库。
- 预览显示每行 `ready / warning / error`，错误行可导出。
- 最终提交使用事务，禁止部分成功。

## 12. 产品库信息架构

产品库主入口改为三个页签：

### 12.1 产品

- 产品卡片和搜索；不再用“零部件产品”承载关键模块。
- 产品详情：当前技术基线、三个关键模块、产品 BOM、关联项目、规格、轻量 Revision。

### 12.2 关键模块

- 按三类模块筛选和搜索。
- 待技术确认、待使用批准、已批准、受限、停用视图。
- 详情：关键参数、内部 BOM、证据、派生链、BOM 差异、where-used、审批记录。

### 12.3 BOM 与配置

- 项目工作 BOM、产品技术基线 BOM、差异和 where-used 入口。
- 批量导入入口与导入历史。
- 普通物料和关键模块通过类型标签区分。

- `products.getCurrentTechnicalBaseline/releaseTechnicalBaseline`
- `bom.prepareImport/validateImport/commitImport`

DRV 创建继续使用单一项目创建接口，输入包含可选 `sourceProductId` 和完整六模块数组。接口负责不变量、快照、BOM 和任务的原子写入。

搜索接口不得返回无权限用户不可见的供应商价格、成本或受限附件。where-used 遵循现有产品/项目访问控制。

## 14. 验收测试

### 14.1 模块主数据

- 工程师可以创建草稿并维护内部 BOM。
- 缺少内部 BOM 或必填参数时不能技术确认。
- 未技术确认不能批准。
- 批准后更新内部 BOM 被服务端拒绝。
- 从批准模块派生会生成新编号并保留来源；原模块不变。
- 增加二供不会创建新模块编号。

### 14.2 DRV 创建

- 来源产品为空时可直接搜索并选择物理关键模块。
- 来源产品有三个模块时全部正确预填；缺失项保持不复用。
- 用户覆盖预填模块后保存精确覆盖结果。
- 六模块零复用只在提交时弹窗；服务端直调同样拒绝。
- 切换 NPD 保留基本字段并清除 DRV 数据。
- ID/CMF 不复用、结构/模具复用时，前端自动修正且服务端拒绝绕过。
- 物理模块复用但无 `approved keyModuleId` 时拒绝。

### 14.3 原子性与任务

- 项目模块基线、工作 BOM、任务或日志任一写入失败时无残留项目。
- 被复用物理模块的 BOM 行与基线模块 ID 一致。
- 来源产品后续换模块不影响已有项目快照。
- 模块从不复用改为复用的组合测试只减少该模块任务包，公共任务不变。

### 14.4 BOM 导入和权限

- 映射、重复、错误行和原子提交均有测试。
- 普通物料不被误建为关键模块。
- 外部或无商业权限账号看不到成本、供应商和受限资料。

### 14.5 可用性

- 模块搜索可完全使用键盘完成。
- 弹窗打开后焦点进入标题/首个操作，关闭后回到创建按钮。
- 所有图标按钮有可访问名称，错误不只依赖颜色表达。

## 15. 分阶段交付

1. **基线整理**：先落地动态项目任务模型，冻结项目类型和六模块语义。
2. **数据与不变量**：关键模块、内部 BOM、技术基线和项目模块基线。
3. **模块管理**：创建、技术确认、批准、派生、限制、搜索和 where-used。
4. **DRV 创建减负**：来源产品可选、模块搜索、提交时零复用引导。
5. **原子联动**：模块基线、工作 BOM 和任务一次提交。
6. **PLM 信息架构**：产品 / 关键模块 / BOM 与配置。
7. **批量导入**：Excel/CSV 暂存、校验和原子提交。
8. **技术基线发布**：项目成果落产品技术基线，轻量 Revision 保持独立。

每一阶段必须能单独上线并有开关/回退路径；在新模块库具备可用数据前，旧自由文本入口只作为短期只读展示，不继续产生新数据。

## 16. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 模块编号爆炸 | 只有受控子件或接口变化才新编号；二供独立管理；提供派生与差异视图 |
| 普通物料污染模块库 | 只允许三种模块类型；导入只生成候选草稿；必须经过两步确认 |
| 初始模块库为空 | 先从测试产品/已验证 BOM 建立少量种子模块，再开放 DRV 强引用 |
| 自动带出造成误选 | 来源产品只是默认值；用户可覆盖；提交前显示模块编号摘要 |
| 模块基线与 BOM 漂移 | 单事务写入、唯一约束、服务端一致性测试 |
| 旧 Revision 语义残留 | UI、API 和文档统一改名；新增技术基线实体，不复用旧 Revision 发布入口 |
| 大工作线冲突 | 先落地动态任务模型，再在同一最新基线上实施；禁止同时改两套创建页 |
| 导入格式不稳定 | 暂存层 + 映射模板 + 行级错误，不直接写生产表 |

## 17. 暂缓决策

以下由管理员配置，不改变本规格核心不变量：

- 模块编号格式和号段。
- 三类模块参数模板。
- 技术基线展示编号规则。
- 第一批 BOM 导入模板及列别名。
- 审批提醒时限和代理规则。
