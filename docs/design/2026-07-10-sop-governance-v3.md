# SOP 治理整改 v3（Sprint 0–3 实施基线）

> 状态：已实现 Sprint 0A / 0B / Sprint 1 / Sprint 2 / Sprint 3 / 盲点收口批次
> 日期：2026-07-10

## 1. 本轮决策

1. 项目轴保留发布后的 2–8 周稳定期，但不承接长期量产治理。
2. Release 与 Close 是两个独立动作：Release 生成量产 Revision；只有 Close Gate 才归档项目。
3. 当前 SOP 模板版本为 `2026-07-v2`。迁移前项目固定为 `2026-07-v1`，任务 ID、阶段内容和完成记录不追溯变化。
4. 每个当前模板 Close 阶段统一使用 `project_close_review`，不再以 CIP、售后、ECO、OTA 或 EOL 常驻任务作为关闭评审挂载点。

## 2. 风险识别

建项或变更范围更新时，填写结构化声明：

- 电芯变化
- 电池包/BMS/保护板变化
- 保护参数变化
- 功率、温升或连续工作边界变化
- 受压结构变化
- 新增目标市场
- 关键安全件供应商/二供变化
- 安全相关软件变化
- EOL 测试变化
- 其他安全或法规变化

规则引擎仅使用上述声明和产品目标市场等结构化字段，不扫描备注、任务描述或其他自由文本。系统可自动将安全/法规风险升为 `high`，普通项目编辑不能降级。

Sprint 1 已增加结构化证书台账。证书覆盖或复用证据不足会自动加入法规高风险原因；只认可有效、未过期且具有证书编号或受控证据引用的记录。Revision 复用还必须明确“已批准复用”和复用边界依据。

## 3. 证书台账与覆盖

系统按变更范围生成确定性要求：

- 电芯/电池包变化：UN38.3、MSDS、电芯/电池包安全认证；
- 保护参数、功率热边界、受压结构或安全软件变化：产品安全认证；
- 关键安全件供应商/二供变化：关键安全件认证；
- 新增目标市场：逐市场准入覆盖；
- 其他安全/法规变化：法规符合性证据。

覆盖范围分为本项目版本、产品族覆盖和 Revision 复用。证书缺口会在项目风险页和 Close Gate 同时显示。

## 4. Gate 会签

每个轨道、每个阶段显式定义七个槽位：产品、研发、QA、SCM、NPI、认证、客户；每槽位为必签、条件签或不适用。

- Gate 轮次开启时，完整冻结会签要求、风险结论和 SOP 模板版本。
- 项目级允许加签，只能从“不适用 → 条件签 → 必签”提高，不能降低。
- 风险上升或新增加签时，当前未完成轮次作废，按新矩阵重开一轮。
- 风险下降不修改已开启轮次，只能影响未来轮次。
- 管理层只能在所有必签槽位完成后作出 Gate 决策。
- 有条件通过必须同时填写条件、负责人和截止日期，并自动生成受控条件项。

## 5. 条件项与让步

- Gate 有条件通过、Release 强制发布和手工让步共用 `project_conditions`，不再形成两套追踪。
- 既有 `mp_releases.followUpOwner/dueDate/conditionsSnapshot` 保留为发布时快照，并链接统一条件项。
- 延期只改变截止日期，状态仍为未关闭，不能用无限续期绕过 Close。
- 只有“关闭”或“转入同产品 ECO”才算闭环。
- 每条未关闭条件项同步为负责人行动项，闭环时同步关闭。
- 历史有条件 Gate 和历史强制发布会迁移为未关闭条件项，避免静默遗漏。

## 6. Close Gate 硬卡

当前模板的 Close Gate 批准前必须同时满足：

- 量产 Revision 已发布；
- 至少两期结构化稳定记录经 QA 确认；
- 记录累计覆盖不少于 14 个自然日；
- 每期产量、FPY 和产能达成目标；
- 无未关闭 P0/P1；
- 结构化证书覆盖完整；
- 所有条件项/让步已关闭或转入同产品 ECO；
- Close Gate 必签槽位已完成。
- 正式量产移交单已由产品维护责任人本人接收。

稳定记录采用结构化数据表，不以“上传两个文件”代替有效校验。Close Gate 不允许有条件通过。关闭成功后产品进入 `maintenance`，项目归档；两者不再共用一个动作。

## 7. 关闭移交与产品运营入口

Sprint 2 已将项目轴与产品维护轴之间的交接棒落成结构化流程：

- 项目团队编制正式移交单，指定产品维护责任人与售后责任人；
- 固定清单覆盖受控资料、维护边界、售后流程和 ECO 流程，每项都必须提供证据引用；
- 提交后生成“量产移交接收”行动项，只有指定维护责任人本人（或系统管理员）可以接收；
- 接收后的移交单一旦被修改，接收状态自动失效，必须重新提交和接收；
- 产品库增加轻量售后问题台账，按已移交的售后责任人自动分派；
- 产品库可从当前已发布 Revision 直接发起 ECO，并强制填写结构化变更范围；高风险规则继续自动生效；
- 售后问题可关联到新建 ECO，保留从客户/现场问题到量产变更项目的追溯链。

Close Gate 增量硬卡只对当前 SOP 模板生效，历史模板不被追溯篡改。

## 8. 产品轴经营与生命周期治理

Sprint 3 将项目费用、轻量软件发版和产品停产治理落在各自正确的责任轴上，不把长期量产运营重新塞回已关闭项目。

### 8.1 项目费用

- 项目可登记模具、认证、NRE、打样、差旅和其他项目性支出；每笔记录预算、实际/承诺金额、币种、负责人和凭证引用；
- 实际支出和已付款记录必须提供凭证引用，已形成实际支出的记录不能硬删除；
- 金额以货币最小单位存储，汇总按币种分别计算预算、实际和偏差；
- 组合看板增加预算偏差列，多币种项目明确显示“多币种”，禁止把不同币种直接相加；
- 商业费用只对管理员、项目负责人/经理、PM 和 SCM 等授权角色开放。

### 8.2 轻量软件 / OTA 发版

- 轻量发版只服务于不影响安全、不改变 BOM 和制造过程的软件更新；
- 只要涉及安全保护、BOM 或制造影响，系统拒绝轻量发版并要求转入 ECO；
- 发版单记录基线 Revision、目标版本、回归证据、灰度方案、回滚方案和独立 QA 责任人；
- 创建人不得自我验证，管理员也不能绕过职责分离；
- 灰度比例只能单向增加，发布后仍保留结构化回滚动作和完整审计记录；
- 产品已进入 EOL 后禁止再创建或推进软件发版。

### 8.3 产品 EOL

- 产品轴增加停产计划，覆盖客户通知、末次采购、库存处置、供应商停供、售后备件承诺、证书归档和替代产品策略七项清单；
- 计划负责人和批准人必须分离，创建人不得自批，管理员也不能绕过；
- 批准后执行清单独立维护，修改执行证据不会破坏已完成的批准记录；
- 完成 EOL 前硬卡日期顺序、末次出货日、七项证据、未关闭项目、P0/P1 售后问题和在途软件发版；
- EOL 完成后产品生命周期更新为 `eol`，禁止再从该产品发起 ECO 或轻量软件发版；
- 软件发版与 EOL 的每次状态变化均写入不可变产品治理事件，保留操作人、时间和状态快照。
