## WF-01 新品全生命周期主干（CRT-01 / GATE-01 / GATE-03 / DTL-01 / DATA-01）

### 结论
- happy path **走通**：建 NPD→自动生成 7 阶段(概念/规划/设计/EVT/DVT/PVT/MP)+各阶段 Gate→逐阶段过 6 个 Gate→currentPhase=mp。刷新后阶段不回退、6 条 Gate 评审历史与 6 个 gate task 完成态全部持久化（DB 核对一致）。

### 问题
1. **【严重】Gate 通过的多写非原子 → 偶发部分持久化 → 阶段被锁死、项目卡死**
   - 现象(n-2IYW6F)：刷新后紧接着过「规划」Gate，结果 `project_gate_reviews` 记下 planning=approved、`currentPhase` 推进到 design，**但 gate task `p7`(Kickoff会议) 仍为 todo**。导致「设计阶段」判定为锁定（提示"请先完成规划阶段 Kickoff会议"），规划进度回显 0%，且**锁定阶段不渲染任何任务勾选框**，无法继续推进。
   - 对照(VdDynTbM 无刷新、tab 切换流程)：concept/planning/design/evt/dvt/pvt 6 个 Gate 的 gate task 全部正确标 done。说明并非必然失败，而是**刷新后/快速连续操作时**触发。
   - 根因方向：一次 Gate 通过要写三处——`projects.update`(currentPhase)+`tasks.setCompleted`(gate task)+`gateReviews.create`——三者**非事务**，由 `handleUpdate`(client/src/pages/Home.tsx:240) 的 **600ms 防抖 + 共享 `saveTimerRef` + clearTimeout** 串起；快速连续 onUpdate 会取消上一笔待写，且 diff 基线取自防抖闭包内的 `project`，刷新后数据未稳定时易丢 task 完成态。network 抓到两笔近乎同时(±23ms)的 200，疑似竞态/双发。
   - 恢复路径：可回到该阶段 tab 重新勾选 gate task→重审→补标 done→解锁；但 Gate 历史已显示 approved 却要求"再审一次"，体验割裂。
   - 优化：把「Gate 通过」做成**一个服务端事务**(写 review + 标 gate task done + 推进 currentPhase 原子完成)；或客户端 gate 操作绕过防抖直接 await 三个 mutation 串行成功后再 invalidate；锁定阶段提供"补登 Gate 结论"的显式入口而非要求重审。

2. **【一般/治理】Gate 就绪度未满足仍可直接「通过」**
   - 现象：就绪度面板明确"还差 9 项 / 不能过会"(缺 5 前置任务 + 缺 4/4 交付物)，评审弹窗的「通过」按钮仍可点，确认后正常推进。4 维就绪检查沦为纯提示。
   - 优化：要么「无条件通过」在硬卡(缺交付物/未关闭P0P1)时禁用、强制走「有条件通过」并留痕；要么就绪度明确标注为"仅建议"。当前介于两者之间，易被赶工的人一路放行。

3. **【轻微】闸口通过后详情页头部「当前阶段」存在短暂不刷新**
   - 现象：confirm 后 ~1.6s 读取头部仍显示旧阶段，刷新后正确。疑似 react-query invalidate 后头部回填有延迟（未稳定复现，待确认）。

## WF-02 闸口不通过与回滚整改（GATE-04 / GATE-05 / RISK-08）

测试项目：`[TEST]闸口回滚-01`(Abl_EpXg)。

### 结论
- 「未通过→整改→重审通过」路径**走通**且可追溯（DB 核对）：
  - 未通过(round 1)：写 `decision=rejected`+原因(notes)，gate task `c6` 置为 **blocked**，currentPhase 不变(停在 concept)。
  - 重审(round 2)：经"重新评审"入口选「通过」，写 `decision=approved`、`roundNumber=2`，c6→done，currentPhase 推进 planning。两轮评审都保留在历史里。

### 问题 / 与测试包预期的偏差
4. **【设计偏差/需澄清】不存在"打回到上一阶段"语义 → RISK-08「交付物随阶段回退」无从触发**
   - 测试包 WF-02 假设"DVT 不通过→项目回退到上一阶段→交付物状态一致回退"。实际模型是：**未通过 = 停留在本阶段、本阶段 Gate task 置 blocked、等整改后重审**，currentPhase 既不前进也不回退；系统没有任何"回退/打回上一阶段"的动作，故也没有"交付物跟着回退"这件事。
   - 评估：这本身是一种合理的 stage-gate 模型，但与测试包/用例集(RISK-08)的预期不一致。建议二选一：① 文档/用例对齐为"停留本阶段重审"语义；② 若业务确实需要"打回上一阶段"，需新增显式回退动作并定义交付物/风险联动回退规则。

5. **【一般/治理】重审时同样可在"不能过会"(还差10项)状态下直接「通过」**——与 WF-01 问题 2 同源，赶工可一路放行。

## WF-03 风险爆发闭环（RISK-01/02/03/05/08 / DASH-08）

测试项目：`[TEST]新品主干-02`(VdDynTbM)，自动健康度=黄灯。

- 厂长(管理层/exec 总览)「需关注项目」列表即时出现该项目 = **手动覆盖:红灯 + 完整原因文案**，无需进项目即可见（DASH-08、RISK-05 下钻看原因 ✓）。红灯计数 4→5。
- PM「恢复自动」→ DB override 清空 → 刷新后回落黄灯自动；厂长看板红灯计数 5→4，该项目从需关注红名单移除，**无残留**（RISK-05 resolution ✓）。

### 问题
6. **【一般/权限】厂长「全局风险看板/管理层视角」仅对 system admin 开放，非 admin 的厂长/管理层看不到大盘**
   - 代码：`OverviewPage` 的 exec 视角硬性 `user.role === 'admin'` 才出现(client/src/components/views/overview/OverviewPage.tsx:28,32)。
   - 实测：以真实 PM 账号 test_pm 登录，视角下拉**只有「PM视角 / 我的视角」，无「管理层视角」**，也不渲染健康度分布/度量对比大盘。
   - 影响：项目成员表里 `manager`(管理层/决策层)、`mfg`(厂长) 这些业务"厂长"角色，若不是系统 admin，则**完全看不到 WF-03/WF-10 依赖的全局风险看板**——厂长决策视角实际等价于"系统管理员视角"。建议：把 exec 大盘授权从"system admin"改为"system admin ∪ 任一项目 manager/mfg"或独立的组织级角色，避免把决策可视性与系统管理员身份绑死。

7. **【轻微，复现】手动覆盖保存/解除后，详情页头部健康度 badge 不即时刷新（需手动刷新页面）**——与 WF-01 问题 3 同源，DB 已正确，仅前端 badge 回填滞后。

## WF-07 多角色协同同一项目（DASH-01/02/05 / PERM-01 / NOTI-03）

测试项目：`[TEST]新品主干-02`(VdDynTbM)，成员 test_pm(pm)/test_ee(rd_hw)/test_qa(qa)/test_scm(scm)。
指派：c2→test_ee、p1→test_qa、d1→test_scm。真实账号登录切换实测。

### 结论：核心协同**走通**
- **千人千面-工作台(按 assignee)**：以 test_ee 登录，「我的任务」只出现指派给它的 c2，**看不到** test_qa 的 p1 / test_scm 的 d1（DB 核对 `getMyTasks` 仅按 `assigneeUserId` 过滤，server/db.ts:2301）。✓
- **千人千面-项目内(按 visibleRoles)**：test_ee 进项目任务清单，概念阶段明确提示"已按岗位过滤，显示 2/6，隐藏 4 项"，只见 C4+C6。✓
- **动作回流**：test_ee 完成 C4 → DB `status=done, updatedBy=5628(研发), completedAt` 写入；切回 test_pm，PM 端任务清单看到 C4「已完成」，且 PM 不受岗位过滤（看到全部 6 项，含对研发隐藏的 c2）。✓（DASH-01/02）

### 问题
8. **【一般/集成】"任务指派(assigneeUserId)" 与 "岗位可见(visibleRoles)" 相互独立 → "被指派但本岗位不可见"的死角**
   - 复现：把 PM 岗位任务 c2(用户需求收集) 指派给 test_ee(rd_hw)。test_ee 工作台「我的任务」能看到 c2 并可点击；但点进去落在项目任务清单，c2 因 `rd_hw ∉ visibleRoles` 被岗位过滤**隐藏**，研发在项目内根本看不到、也无法打开/完成这条被指派给自己的任务（深链 `taskId=c2` 也只停在过滤后的列表，不展开）。
   - 影响：跨岗位指派(常见：PM 把某事临时丢给研发)会产生工作台可见、项目内不可达的断点；用户只能在工作台层操作（若工作台无 inline 完成入口则彻底卡住）。
   - 优化：① 被指派人对"指派给自己的任务"应无条件可见（assignee 优先于 visibleRoles 过滤）；② 或工作台深链时强制展开该任务详情抽屉，绕过列表岗位过滤；③ 或指派时校验目标用户岗位在该任务 visibleRoles 内，否则提示。

9. **【未覆盖】NOTI-03 指派通知 / 采购物料待办**：本轮指派经 DB 直配，未走 PM 的 UI 指派流程，故指派通知(NOTI-03)未实测；采购(scm)视角 BOM/物料待办(DASH-04 类)未深入。建议后续补测。

## WF-09 异常恢复流（DATA-03 / EXC-02 / EXC-03）

测试方式：拦截 `fetch` 让 trpc 保存 mutation 返回 401（模拟会话过期/断网），编辑「客户」字段并失焦触发保存。

### 结论
- **不静默丢失、不假成功**(EXC-03 ✓)：保存失败时状态变「同步失败」，编辑值仍留在输入框；恢复网络后重新失焦即成功保存（"已同步至云端"，DB 落库 = 恢复后的值）。

### 问题
15. **【一般/体验】保存失败提示太弱、无重试/重登引导（EXC-02 不达标）**
    - 失败时**只有侧边栏底部一行小字「✕ 同步失败」**，无 toast、无弹窗、无"会话已过期请重新登录"、无"重试"按钮；窄屏侧边栏收起时该提示完全不可见。401(会话过期) 与普通网络错误被一视同仁，不会引导重新登录。
    - **无草稿持久化**：未提交内容仅存在 React state，若失败后用户刷新/关页，编辑内容即丢失（无 localStorage 草稿）。测试包要求的"恢复后未提交内容有草稿/提示"不满足。
    - 优化：保存失败弹 toast + 顶部常驻条 + 重试按钮；401 单独处理（提示重新登录并保留草稿）；对编辑中表单做本地草稿暂存。

## WF-10 厂长决策视角全景下钻（VIZ-01/03/04 / PERM-02/03）

### 结论
- **分布图准确 + 下钻顺畅**(VIZ-01/03/04 ✓)：健康度分布 红4(22%)/黄10(56%)/绿4(22%)=18、阶段分布合计 18，均与 DB 一致；从需关注红名单点入 → 进红色项目详情(qa-login-project) → 风险原因(UN38.3 覆盖原因)清晰可见。
- **只读角色编辑被拦截**(PERM-03 ✓)：以 test_scm(scm, canEditProjectInfo=false) 登录看项目，详情显示「仅可查看」、无「量产发布」按钮、PM 选择器禁用——非编辑角色的写操作被正确 gate（前端隐藏 + 后端 ROLE_PERMISSIONS 双重）。

### 问题
16. **【一般/权限设计】没有"只读厂长/决策层"的全景视角路径（PERM-02 设计矛盾）**
    - 全局风险/阶段大盘只有 system admin 能看（见问题6），而 admin 在任意项目都是**满权限**（实测 admin 进 qa-login-project 可见量产发布、健康度覆盖、改 PM——完全不只读）。真正只读的 `viewer`/`mfg` 角色又看不到大盘。
    - 结论：测试包设想的"厂长=只读决策者，看全景但不能改"在当前模型下无法实现——要看全景必须是 admin（能改一切），只读角色看不到全景。建议为"决策层只读"设计独立角色（可见 exec 大盘 + 全局只读下钻），与系统管理员解耦。
