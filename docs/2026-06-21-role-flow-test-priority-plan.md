# 角色化流程测试结果整合与优先级方案

日期：2026-06-21

## 结论先行

当前角色与流程底座是稳定的，不建议重做角色模型。全量回归已通过：52 个测试文件、337 条测试全部通过。角色相关的重点测试也已通过：PM 自动入组、按角色派工、权限不降级、PM 提交流程裁剪、admin 审核、PM 工作台排序等 30 条用例全部通过。

下一步的重点不是补"能不能用"，而是补"不同角色每天打开后能不能顺手完成工作"。优先处理顺序建议为：

1. Portfolio 度量 rollup：管理层先能横向看项目效能。
2. 交付物类型与版本：执行角色上传/复核的核心信息补齐。
3. 需求价值链路：把商业目标、项目目标、需求、任务串起来。
4. 异常升级机制：把逾期、阻塞、P0/P1 从提醒升级到闭环治理。
5. 风险生命周期：从项目 risk 字段升级为可跟踪风险列表。
6. 角色分派升级：只做立项补角色清单和提醒，不重做已有自动派工。

## 已验证的角色流程

### 1. 权限与角色解析

已验证点：
- PM 用户不会把已有 manager 角色降级为 pm。
- 当成员角色较低时，pmUserId 会补成 pm。
- 同时是 pmUserId 的 manager 仍可创建 Gate 评审。
- admin/owner/manager/pm/执行角色的项目访问和操作权限已有统一入口。

结论：角色层级和项目访问底座可信，可以作为后续所有流程的基础。

### 2. 立项、PM 入组、按角色派工

已验证点：
- 立项后 PM 可被幂等加入项目成员，不覆盖既有角色。
- 任务可按 visibleRoles 自动分配给对应角色成员。
- gate 或无角色任务会落到 PM。
- 已经手动分配的任务不会被自动派工覆盖。

结论：P3"角色分派升级"不应再做成大改，剩余价值主要是建项目后的"待补角色清单"和提醒。

### 3. 产品定义交接到角色任务

已验证点：
- 创建 NPD 项目不强制要求先关联产品库。
- 关联已确认产品定义时，会锁定产品定义快照。
- 产品定义交接能按角色分桶，并生成对应任务。
- 生成的任务会带正确 visibleRoles，例如结构/ID 规格进入 rd_mech 任务。

结论：产品定义到角色任务的主链路已经打通，后续更值得补的是需求价值链路和交付物版本化。

### 4. PM 工作台

已验证点：
- PM 视角只保留自己负责的项目。
- TODAY 卡纳入逾期/今日任务、本周 Gate、红灯或延期风险项目。
- 待协调/拍板卡按"待审交付物 > P0/P1 > 未分配 > 交付物缺口 > Gate 未就绪 > 阻塞"排序。
- 我负责的项目优先展示 P0/P1，其次逾期，再其次阻塞。

结论：PM P0 三卡的排序逻辑已经可用。下一步若继续优化工作台，优先做 UI 走查和角色入口，而不是新增复杂服务端框架。

### 5. 裁剪、Gate、交付物、发布门禁

已验证点：
- 非 PM/非 admin 不能提交流程裁剪。
- PM 可以提交裁剪，admin 可以审核。
- 被裁阶段会从日历过滤，Gate 就绪会短路。
- Gate 就绪覆盖前置任务、交付物、P0/P1、评审条件。
- 交付物审核已接入：提交、通过、驳回、重传回待审、myPending、Gate 口径均有测试。
- MP Release 硬闸已覆盖 P0/P1、交付物、Gate 等条件。

结论：流程门禁不是空壳，已经具备较强治理能力。短期不宜再叠复杂审批，应该补交付物类型/版本这种执行必需字段。

### 6. 自动化、健康度、延期影响

已验证点：
- RAG 健康度、健康摘要、自动化规则、延期影响分析、改期冲击 emit 均有测试。
- 延期重排会跳过被裁剪 skipped 任务。
- 自动化规则和 digest 去重机制已覆盖。

结论：提醒机制可继续复用。下一阶段"异常升级机制"应基于已有自动化引擎加规则，不另起一套通知系统。

## 暴露出的缺口

### A. 管理层还缺横向效能视图

单项目度量已经有，但组合层还不能横向比较项目 Lead Time、逾期率、近 4 周吞吐、Gate 一次通过率。已有 spec，无 plan、无代码，是当前最短路径的高价值项。

### B. 执行角色的交付物信息还不够完整

交付物审核已做，但文件自身缺 fileType/fileVersion。结构、硬件、QA 等角色在上传/重传时，很难表达"这是什么类型、这是第几版、当前审核对应哪版"。这是执行闭环里最容易让人困惑的点。

### C. 需求到任务的价值链路偏薄

现在 projectRequirements 能承接需求池，但商业目标、项目目标、需求、任务之间还缺可追溯关系。后续做 OKR 或项目复盘时会缺因果链。

### D. 异常升级还停留在提醒层

已有逾期、阻塞、P0/P1、健康摘要，但缺"超过多久自动升级给谁"。这会影响 PM 和 manager 的闭环治理。

### E. 风险还是项目字段，不是生命周期对象

项目 risk 能驱动 RAG，但无法记录风险项、状态、责任人、缓解计划和关闭时间。

## 建议解决顺序

### P0：Portfolio 度量 rollup

理由：spec 已就绪，依赖少，能直接提升管理层判断力。

范围：
- 补 implementation plan。
- 新增组合度量纯函数、后端聚合、analytics 路由、总览页对比表。
- 复用单项目 metrics，不改现有 metrics 核心。

验收：
- 管理层能在总览看到项目度量对比表。
- 可按逾期率、Lead Time、吞吐、Gate 通过率排序。
- 新增纯函数测试和 DB 集成测试，全量回归通过。

### P1：交付物类型 + 版本

理由：交付物审核已经上线，fileType/fileVersion 是当前执行角色闭环的关键缺口。

范围：
- project_files 增加 fileType/fileVersion。
- 上传 UI 支持选择类型、填写版本。
- 交付物列表和审核态显示当前版本。
- 重传后明确当前审核对应最新版本。

验收：
- 用户能看出每个交付物的类型和版本。
- 新版本上传会触发重审，页面上能看出"待审的是哪一版"。
- 不破坏现有文件上传、审核、Gate 就绪测试。

### P1：需求价值链路

理由：这是下一块真正提升业务价值的主功能，但会牵动数据模型和 UI，应排在两个短路径收口之后。

范围：
- 商业目标/项目目标/需求/任务建立可追溯关系。
- 需求转任务时保留来源。
- 项目详情和需求池能查看链路。

验收：
- 能从任务反查需求、项目目标、商业目标。
- 能从需求查看下游任务完成度。
- 变更、问题或交付物能逐步挂到需求上下文。

### P2：异常升级机制

理由：已有自动化底座，新增规则的投入产出比高，但要等交付物和度量口径更稳定后做。

范围：
- 配置按时长阶梯升级：负责人 -> PM -> manager。
- 覆盖逾期任务、阻塞任务、P0/P1、待审超时。
- 复用 automation_rules、automation_runs、通知与钉钉通道。

验收：
- 同一异常不会重复轰炸。
- 达到阈值后能升级到上一级角色。
- 处理/关闭后不再升级。

### P2：风险生命周期

理由：风险对象化能补齐项目治理，但比异常升级更偏管理流程，优先级略后。

范围：
- 独立 risk list：状态、等级、责任人、缓解计划、目标日期。
- RAG 可读取风险列表，而不是只读 project.risk。
- 总览和项目详情增加风险入口。

验收：
- 风险可创建、更新、关闭。
- 高风险未关闭能驱动 RAG。
- manager/PM 能看到风险责任人与截止日。

### P3：角色分派升级

理由：自动派工已存在，剩余是立项体验和缺口提示，不该高于价值链路和异常闭环。

范围：
- 建项目后输出待补角色清单。
- 缺关键角色时提醒 owner/PM。
- 自动派工结果增加可解释提示。

验收：
- 新项目能清楚看到哪些角色缺人。
- 补成员后可重新派工。
- 已手动分配任务仍不被覆盖。

## 推荐本周推进节奏

1. 先做 Portfolio 度量 rollup：1 个计划文档 + 1 个实现闭环。
2. 接着做交付物类型/版本：先确认 DB 字段和 UI 命名，再实现。
3. 同步准备需求价值链路 brainstorm/spec，不急着编码。
