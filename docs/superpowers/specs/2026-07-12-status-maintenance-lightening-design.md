# 日常状态维护轻量化 — 设计文档

日期：2026-07-12
状态：✅ 已实现 — 施工计划见 `docs/superpowers/plans/2026-07-12-status-maintenance-lightening.md`
范围：任务状态更新、证据提交、通知体系的交互模型重设计
姊妹文档：`2026-07-12-npd-template-slimming-tiering-design.md`（痛点 A：模板瘦身分档）

## 1. 背景与问题

痛点 C：更新任务状态、填进度、处理通知这类高频小操作太琐碎，加重员工负担。
关键事实（用户确认）：**员工常驻钉钉，很少主动打开网站**。"打开网站→找项目→找任务→点状态"这条路径本身就是负担的主要来源。

## 2. 总方向（已确认：D = 自动化优先 + 钉钉卡片一键操作）

- **自动化优先**：能由系统事件推断的状态变化，不让人动手。
- **钉钉卡片是主操作入口**：剩下必须人做的动作，由钉钉通知卡片发起（▶开始 / ✅完成 / 📎去上传 / ⏰延两天）；▶开始和延后直接回写，轻证据进入一句话完成页，重证据进入任务上传页。
- **网站退居"看全局 + Gate 评审"**：项目看板、依赖链、Gate 签核、重证据上传在网页端。
- 复用已有底座：action-card 机制、工作通知（`dingtalkMessage.ts`）、snooze、以及 6-27 钉钉引擎设计的可靠性/审批底座。系统仍是单一真相源，卡片操作先落系统再生效。

## 3. 证据分级（已确认：A 方案）

任务在模板中标注证据类型（与瘦身设计的任务定义联动）：

| 证据级别 | 定义 | 完成路径 |
|---|---|---|
| **轻证据** | 一句话结论（可包含链接） | 钉钉卡片进入轻量完成页；填写一句话后与"完成"原子回写 |
| **重证据** | 测试报告、设计包、认证证书等文件 | 卡片"完成"按钮变为"去上传"，跳转网页任务页；**由任务责任人自己上传**（非 PM 代传） |

- 上述证据硬门槛对 NPD `2026-07-v3` 生效：轻证据没有非空 `completionNote` 不得完成；重证据没有任务责任人本人上传的任务文件不得完成，且同角色的其他人、PM 或管理员不能代替责任人提交。老模板保留既有完成行为，避免存量项目被新规则卡住。
- 不引入"完成待补证据"中间态；Gate 评审照常硬性检查阶段证据齐套。Gate 任务不提供普通任务的开始、上传后完成或直接勾选完成入口，只能由正式 Gate 评审结果推进。
- 轻量档任务应绝大多数为轻证据（承接瘦身设计的"交付物证据化"精神）。

## 4. 自动化规则（五条，全部确认）

1. **证据与完成合一**：轻证据一句话随完成动作原子提交；重证据网页上传后弹"标记完成？"，服务端再次校验责任人与文件后才完成。
2. **前置完成 → 推"可以开始"卡片**：后继任务状态**不自动变**（仍是待开始），只给责任人推带 ▶开始 按钮的卡片。依赖链驱动的是*通知*，不是*状态*。⚠️ 铁律：前置未完成 = 待开始不是 blocked；blocked 不由依赖图派生。
3. **Gate 通过 → 自动关闭 Gate 任务** + 触发下阶段首批任务的"可以开始"卡片。Gate 签核本身就是完成证据。
4. **到期未动 → 摘要 + 增量升级**：责任人的今日到期/逾期项进入每日个人摘要；逾期 2 天只新增提醒 PM，逾期 7 天只新增提醒管理层；**不自动改状态**，不要求填逾期说明。零散 `due_soon_reminder` / `overdue_reminder` 默认关闭，但保留给管理员显式启用。
5. **"开始"永远由人点**：不做"传了草稿就自动 in_progress"推断，开始是人工承诺。计划排期继续写 `startDate`，人工承诺单独写 `actualStartedAt`；负责人、计划开始日和到期日均不得自动产生 `in_progress`。

开始与完成共用一套服务端执行守卫：任何模板都不得操作未来阶段任务或直接推进 Gate；NPD v3 还必须使用项目分档、附加包和裁剪后的**有效依赖图**，前置任务未完成时既不能开始，也不能从任何网页、卡片或旧勾选入口绕过完成。

## 5. 通知分层（已确认：A 方案）

| 层 | 内容 | 通道与形式 |
|---|---|---|
| **即时（卡片带按钮）** | 仅"新进入收件人处理队列"的：可以开始了、被指派/被@、审批到你了 | 钉钉个人工作通知，交互卡片；点按钮 = 处理完通知，通知即操作入口 |
| **每日摘要** | 早上一条聚合卡片"你今天有 N 个任务要动"，包含今日到期并将逾期置顶，替代零散到期/逾期提醒 | 钉钉个人工作通知，每日一条 |
| **纯知会（FYI）** | "某某完成了 XX"、项目进度类 | 不发个人通知；沉淀到项目群日/周摘要，管理层看周报 |

**防重规则**：责任人的到期/逾期信息只进每日摘要；异常升级每层只通知新加入的角色（day 2 仅 PM、day 7 仅管理层），不重复拉责任人和上一层。

## 6. 与瘦身设计的联动

- 任务模板使用字段：`evidence: 'light' | 'heavy'`；查询层对未知任务与老模板兼容回退为 light，但服务端证据硬门槛仅对 NPD v3 启用。
- 瘦身后人均同时在手任务很少，即时通知量天然可控——两个设计互为前提。
- Gate 检查逻辑改读模块状态（Issue List P0/P1 关闭等），与卡片操作互不冲突。

## 7. 实施要点（拆计划时展开）

1. 卡片动作端点：▶开始 / ✅完成（进入一句话完成页）/ 📎去上传 / ⏰延两天，走 action-card token + 深链机制，幂等、失败可见（复用 6-27 引擎设计的可靠性要求）。
2. 自动化规则引擎接入 `server/automation/`（rules + scheduler 已有雏形）：依赖完成事件、Gate 通过事件、到期扫描（cron）、重证据上传后确认弹窗。
3. 每日摘要 cron + 项目群周摘要；现有零散提醒改造/下线，防重表。
4. 延两天（snooze）只推迟行动项可见性到上海日历后天 08:00，不改任务 dueDate。
5. 通知发送结果统一落 `automation_runs`，失败可重试（6-27 设计的 KR）。

### `task_ready` 生命周期

- 身份使用 `(projectId, phaseId, taskId)` 组合键，避免不同阶段同名任务互相关闭；同一责任人的重复就绪事件只保留一条，且不会清掉用户已经设置的 snooze。
- 任务被显式开始、直接完成、提交审批或审批通过时，关闭对应的 `task_ready`，并把已发送的钉钉互动卡同步标为 handled；Gate 不创建 `task_ready`。
- 任务改派时先关闭旧责任人的 `task_ready`，再按同一有效依赖图重新判断；只有任务仍为待开始、前置全部完成且新责任人存在时，才给新责任人创建卡片。

## 8. 已决事项与外部前置

- "延两天"已按直接生效、仅记录实现，不需要 PM 审批。
- 项目群周摘要固定包含：本周完成、当前逾期及责任人、下周到期、当前阶段与 Gate；上海周一 09:00 每项目群一条，同周原子防重。
- 生产环境需确保钉钉应用具备企业会话 `chat/send` 权限，且项目已保存 `dingtalkChatId`；发送失败会记录错误并在本周后续扫描重试。
- 两次迁移：0063 新增 `completion_note` 与 `actualStartedAt`；0064 扩展 `action_item_kind.task_ready`。
