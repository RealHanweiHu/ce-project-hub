# NPD 模板瘦身与分档裁剪 — 设计文档

日期：2026-07-12
状态：实施中 — 计划见 docs/superpowers/plans/2026-07-12-npd-template-slimming.md
范围：`shared/sop-templates.ts` NPD 模板（55 任务）→ 核心 25 + 附加包 7，轻量档 ≈15

## 1. 背景与问题

系统目的：集中项目信息、跟进状态、促进角色配合、推动项目按时按质完成。
现状痛点（用户确认）：

- **A. 任务和交付物太重**：NPD 模板 55 个任务、40+ 种交付物，很多任务对小项目没意义；每任务都要产文档，小团队（多岗位一人兼任）负担过重。
- **C. 日常状态维护太繁**：高频小操作琐碎（本文档不覆盖，另行设计）。
- **D. 系统尚未大规模推行**：可做破坏性简化，无存量迁移顾虑。

方向决策：**先瘦身、再分档**。先把 55 任务的完整模板砍到核心 25 + 附加包 7，再在瘦身后的模板上裁出轻量档（≈15）。审批/Gate 签核逻辑保留不动。

## 2. 瘦身规则（七条纪律）

1. **任务门槛（四项判据）**：一个动作要成为系统任务，必须至少满足其一——①改变状态（推动项目/对象状态机前进）②明确责任（有独立责任人和截止时间）③形成决策（产生 Go/No-Go 或选型结论）④产生审计证据（留下 Gate/追溯要查的凭据）。四项全不满足的职能动作（调研、沟通、整理、确认），放入操作指南或检查项，不再单独建任务。
2. **过程动作降级**："调研、沟通、整理、确认"等过程动作合并到操作指南或子检查项。
3. **项目设置移出 SOP**：如"团队组建与资源分配"从任务清单删除，保留在立项配置中一次完成。
4. **同一成果只留一个任务**：如市场调研/VoC/概念定义合并为一个立项论证任务，内容作为内部检查项。
5. **同一证据只上传、审核一次**：后续任务和 Gate 直接引用（如包装验证 v6+pv6 合一）。
6. **条件性任务进附加包**：电池、软件、认证、模具等非所有项目都需要的任务，移入附加包，立项时按需激活。
7. **三条红线**：安全/认证、量产发布、客户放行，任何档位不得取消。

**独立工作线不合并**（重要修正）：ID/MD/EE 设计等各有责任人、耗时最长、存在跨线依赖的工作，必须保留为独立任务——合并会使关键路径（结构等外观、Layout 等结构）不可见。合并仅适用于"同一责任人 + 同一份证据"的情形。

## 3. 复杂度预算（已确认）

- **核心档 = 25 个任务**（含 7 个 Gate）
- **常规组合（核心 + 部分包）≤ 30**
- **四包全激活允许到 32**（新电池+新认证+新固件+新模具 = 真正全新 NPD，本应最重）
- 今后新增任务必须先过四项判据（改变状态/明确责任/形成决策/产生审计证据，至少满足其一），再说明为什么不能作为检查项、操作指南或附加包，防止模板回涨到 40~50。

## 7. 自动生成最小必要流程（分档推导，用户追加）

档位与附加包不靠人凭经验选，由系统按项目属性推导，人只确认或有据否决：

**输入**（立项向导采集，多数已是既有字段）：项目类型（category）、产品属性（是否含锂电/受压腔体、含固件/APP、需开新模、目标市场认证需求）、变更范围（ECO/衍生品的 changeScope 声明）、风险等级（`safetyRiskLevel`/`regulatoryRiskLevel` 既有列）。

**推导规则**（纯函数 `recommendNpdTemplateConfig`，shared 层）：

| 条件 | 结论 |
|---|---|
| 含锂电/受压腔体 | 🔋电池包**强制激活且锁定**（红线，不可取消勾选） |
| 有目标市场认证需求（出口/强制认证） | 📜认证包**强制激活且锁定** |
| 含固件/APP | 💾软件包默认勾选（可取消） |
| 需开新模 | 🔨模具包默认勾选（可取消） |
| 安全或法规风险 = high，或 电池+认证双锁定，或全新平台 | 档位 = **强监管**（即完整档 full 的显示名，四包按属性全开，红线四眼默认从严） |
| 无电池、无新模、风险 standard、简单新品/小改 | 档位 = **轻量** |
| 其余 | 档位 = **标准** |

**交互与治理**：

- 向导先答 4-5 个属性问题，实时显示推荐档位+包组合和**推导理由**（"因含锂电 → 电池安全包已锁定"）。
- 人可改推荐（升档随意、降档需填理由），但**锁定包不可取消**——这是红线在立项环节的第一道执行点。
- 否决记录进 `customFields.npdTemplate`（`recommended` vs `chosen` + 理由）并写活动日志，审计可见"谁在什么理由下降了档"。
- 服务端按属性**重新推导并校验**锁定包（不信任前端提交），缺锁定包直接拒绝创建。
- 非 NPD 品类不走本推导：衍生品已有复用等级裁剪机制，ECO/IDR 模板本身就小。
- 档位显示名调整：轻量 / 标准 / **强监管**（full 的对外名称，强调"不是更繁琐，是监管要求"）。

## 8. 连带改动（实施时注意）

1. **Gate 检查逻辑改为读模块状态**（已确认）：e5/mp4 删除后，Gate 4 通过条件改为"Issue List 中 P0/P1 全部关闭"，关闭评审读 Change Log/RMA 数据，而非检查任务完成。
2. **立项配置页扩展**：档位选择（轻量/标准/完整）+ 附加包勾选 + 成员角色指派（承接 p6）+ 排期初始化（承接 p3）。
3. **交付物清单同步瘦身**：44 种交付物按"同一证据只传一次"原则重排到新任务下（实施计划中逐项映射）。
4. **衍生品/ECO/IDR 模板不在本次范围**，但附加包机制与 `DERIVATIVE_REUSE_MODULE_RULES` 应统一抽象，避免两套裁剪逻辑。
5. **新增任务准入**：PR/评审时必须回答"为什么不能是检查项/指南/附加包"，写入模板文件头部注释。
