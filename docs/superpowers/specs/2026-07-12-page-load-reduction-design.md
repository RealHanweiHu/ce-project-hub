# 页面减负与渐进展示 — 设计文档

日期：2026-07-12
状态：🟢 主体已实现（2026-07-13，§1-§6 已落地并经复审整改；未完项明确列于 §10"遗留"，以该清单为准）
范围：项目总览、阶段任务、任务详情、Gate 详情四个页面的呈现层减负 + 项目状态口径统一（数据层） + 个人工作入口合并（"我的工作"三桶） + 四项验收指标
姊妹文档：`2026-07-12-npd-template-slimming-tiering-design.md`（源头减量）、`2026-07-12-status-maintenance-lightening-design.md`（操作出口外移）、`2026-06-25-project-detail-dashboard-design.md`、`2026-06-25-task-detail-redesign-design.md`（既有 UI 改版，本设计在其上叠加，不推倒）

## 0. 原则

三份已确认设计从**源头**减量（任务 55→15/25）、把**操作出口**挪到钉钉；本设计解决最后一段：留在网页里的信息如何呈现。统一原则：

> **每个页面首屏只回答一个问题。** 总览答"项目卡在哪"；任务列表答"我现在该做什么"；任务详情答"这个任务怎么算完成"；Gate 详情答"还差什么"。其余信息一律折叠、后置或链走。

## 1. 项目总览（用户反馈 #3）

现状：三栏仪表盘并列 待办任务/未关闭问题/关键信息/最近变更 + 头部阶段步进器，信息密度高但无主次。

改为**焦点三卡置顶**，现有卡片整体下沉为次要区：

| 卡 | 内容 | 数据源（均为既有查询） |
|---|---|---|
| 🎯 下一里程碑 | 下一个 Gate 名称 + 目标日期/剩余天数 + 缺口数（"距 Gate 3 还差 4 项"） | `getGateReadiness`（当前阶段） |
| 🚧 前三阻断 | 按严重度取 3 条：未关闭 P0/P1 → 逾期红线任务 → 缺失必交证据；每条带责任人与"催办"按钮 | readiness.criticalIssues + incompleteTaskIds + 缺失 requiredDeliverables |
| ⏳ 待决事项 | 等当前用户（或 PM 视角等任何人）动作的审批/签核/审核，每条一键跳转 | action_items（pending_approval / deliverable_review / gate signoff pending） |

- 三卡排版固定（沿用 6-25 设计的"排版不随内容增减改变"约束），空态显示"无阻断 ✓"。
- 头部阶段步进器保留（它就是里程碑语境），其余卡片折到"更多"区或下滚可见。

## 2. 阶段任务（用户反馈 #4）

现状：进入任务 tab 看到全量任务平铺，先感受到流程规模而不是"我该做什么"。

- **默认视图 = "我的·当前阶段"**：只展开当前阶段，默认过滤"指派给我"，排序 = 可开始（前置已清）→ 进行中 → 按截止日。其他阶段收成手风琴条（只显示"设计阶段 4/6 ✓"进度摘要），点开才见任务。
- PM/管理层默认视图 = 当前阶段全员任务，同样手风琴；"全部阶段展开"是显式动作不是默认。
- 列表/看板/甘特的视图切换保留（6-25 设计），本条只改**默认打开姿势**。
- 源头减量由瘦身计划负责（轻量档 15 个）；本条保证即使标准档 25+包，员工首屏也只见自己当下的 3-5 个。

## 3. 任务详情（用户反馈 #5）

现状：交付物、Issue、说明、评论、活动、流转、审批、附件分类、可见范围同屏。6-25 设计已改两栏+4 标签，本设计在其上定**渐进展示规则**：

**首屏（主栏）按"角色 × 状态"只渲染当下相关的：**

| 状态 \ 视角 | 责任人 | PM/管理层 |
|---|---|---|
| 待开始 | 操作指南 + ▶开始 + 前置任务状态 | 同左 + 改期/改派 |
| 进行中 | 证据槽（轻：一句话输入框；重：上传按钮）+ ✅完成 + 本任务关联 Issue | 同左 + 催办 |
| 待审批/已完成 | 完成结论 + 已交证据（只读） | 审批操作 / 复核 |

- 评论/活动/流转/审批四标签保留但**默认折叠**（6-25 的 4 标签区），不抢首屏。
- 附件分类、可见岗位、审批闸门开关等**配置类字段**收进右属性栏的"更多设置"折叠区，仅 PM/管理层可见——员工视角永远不出现。
- 渐进展示是**渲染规则**不是权限变化：数据可见性仍由既有 visibleRoles/权限控制。

## 4. Gate 详情（用户反馈 #6，高优先级）

现状：同一份交付物在任务清单、资源管理、审核状态、Gate 标准四处重复出现，员工要交叉核对。

**收敛为一张"缺口清单"（Gap List）**，即 `getGateReadiness` 结果的完整渲染——它本来就在服务端算齐了所有缺口，只是前端没把它当唯一视图：

```
Gate 3 设计冻结 —— 还差 4 项
──────────────────────────────
□ 任务未完成   PCB Layout            @王工   [催办]
□ 证据缺失     BOM v1.0              @采购   [去上传]
□ 审核未通过   结构 3D 设计（待审）    @QA    [去审核]
□ P1 未关闭    #123 电池仓干涉        @结构   [看 Issue]
✓ 签核就绪     产品/研发/QA 槽位已配置
──────────────────────────────
[发起 Gate 评审]（缺口清零前置灰，hover 说明差什么）
```

- 每行 = 一类缺口 + 责任人 + 一键动作（动作直接复用状态维护计划的深链/卡片端点）。
- 任务清单、资源管理、审核状态页里的交付物**改为链接到缺口清单对应行**，不再各自渲染一套完成度——"同一证据只出现一次"从存储层（瘦身设计规则 5）贯通到呈现层。
- Gate 标准（entry/exit criteria）折叠为缺口清单头部的"评审标准 ▸"，展开才见全文。

## 5. 统一项目状态口径（数据层，用户追加）

现状核实：**"进度"存在三套并存算法**——①客户端 `computeOverallProgress`/`computePhaseProgress`（`client/src/lib/data.ts:314+`，项目列表/甘特/项目详情三处调用），依赖前端水合的 `project.phases[].tasks` 形状，**列表页轻量对象缺阶段数据时恒显 0%**；②服务端 `getPortfolio`（`server/db.ts:2196`）SQL 聚合的健康度/计数；③`projects.progress` 存量列。三套口径互不通气，模板 v3 分档上线后客户端重算还会因不知道裁剪配置而系统性算错。

**目标状态：**

- 服务端提供唯一的 `getProjectStatusSummary(projectId)`（或批量版），一次算齐并返回：`progress`（按生效任务集，skipped 除外）、`phaseProgress[]`、`health`、`currentPhase`、`blockerCount`（未关闭 P0/P1 + 逾期红线）、`nextGate`（名称/日期/缺口数）。内部复用 `getEffectivePhasesForProjectLike` + `getGateReadiness` 的既有构件，与缺口清单同源。
- **项目列表、总览、搜索、项目详情、甘特、周报/摘要全部只消费这一个结果**，不再各自重算。
- 客户端 `computeOverallProgress`/`computePhaseProgress` **删除**（不是保留做兜底——兜底就是第二套口径的复活通道）。
- `projects.progress` 列降级为派生缓存：在 `refreshProjectTaskStatuses` 末尾顺手回写，仅供排序/导出等不便实时聚合的场景，任何页面展示不直接读它。
- 一致性由测试锁死：对同一项目断言 列表接口进度 = 详情接口进度 = 状态摘要进度（对应 §7 指标四）。

## 6. 合并个人工作入口："我的工作"（用户追加）

现状核实：个人工作散在两个入口——"我的任务"独立页（`MyTasksView.tsx`）与总览"我的工作台"面板（`PerspectivePanel.tsx`，另有质量/NPI/工程三个角色变体），员工要两头看。但**聚合层已存在**：`workbench.mine`（`server/routers/workbench.ts:59`）已一次返回任务/行动项/角色任务/待审核/Issue/Gate 阻断；@提醒有评论 `parseMentions` 数据源；分桶逻辑 `buildWorkbenchQueue` 目前长在客户端。

**目标状态：一个"我的工作"页，三个桶，取代两个入口。**

| 桶 | 判定 | 内容 |
|---|---|---|
| 🔥 现在处理 | 下一步动作在我 | 可开始/进行中/今日到期的我的任务、等我决定的审批、指派我的审核、待我签的 Gate 槽位、指派我的 P0/P1、未读 @提醒、开放行动项 |
| ⏳ 等待别人 | 我已出手，球在别人那 | 我提交的待审批任务、我传证据待审核的交付物、我的任务卡在前置未完成（显示"等 @某人 的 XX 任务"——注意语义仍是待开始不是阻塞）、我报的 Issue 待修复 |
| 👀 仅关注 | 无需动作的知会 | 我参与项目的近期动态、被 cc 的评论、已 snooze 项（含恢复时间） |

- 分桶器下沉到服务端（`workbench.mine` 返回结构直接带桶标签），与钉钉每日摘要共用同一分类——网页"我的工作"和钉钉摘要卡是同一份数据的两个出口，天然一致。
- `workbench.mine` 需补两类源：待我签的 Gate 槽位（从 readiness 签核槽 pending + 我的角色匹配）、未读 @提醒（评论 mentions）。
- "现在处理"每行带一键动作（▶开始/✅完成/去审批/去签核），复用状态维护计划的深链与端点。
- **"我的任务"页与总览工作台面板下线**，路由重定向到"我的工作"；质量/NPI/工程角色变体降级为"我的工作"内的可选过滤器（管理视角保留总览的组合健康度，那是项目视角不是个人视角，不在合并范围）。
- 排序：现在处理内按 逾期 > 今日到期 > 可开始 > 进行中；跨项目混排，行内带项目徽标。

## 7. 验收指标（用户指定四项）

| 指标 | 度量方法 | 数据源 | 目标方向 |
|---|---|---|---|
| 员工每周维护系统时间 | 人均每周"维护类操作"次数 ×估时（状态改动/进度填写/通知处理），钉钉卡片一键处理单独计数应占比上升 | activity_logs + automation_runs 按周聚合 | 显著下降；卡片处理占比 >60% |
| 单阶段人工操作次数 | 一个项目从阶段开始到 Gate 通过的人工写操作数（状态/上传/编辑/审批） | activity_logs 按 (project, phase) 聚合 | 轻量档目标 ≤ 每任务 3 次 |
| 证据重复上传率 | 同一 (project, deliverableName) 上传 >1 次的比例（剔除审核驳回后的重传） | project_files 聚合 | <10% |
| 不同页面状态一致率 | 缺口清单成为唯一事实源后做抽样比对：总览阻断数 / 任务列表完成数 / Gate 缺口数三处交叉一致 | 自动化一致性测试（对同一项目三个查询断言互算相符）+ 上线后抽检 | 100%（靠单一数据源保证，不靠人肉对齐） |

前两项需要一个轻量的**指标看板页或周报**（管理层视角），随本设计实施；建议先跑 2-4 周基线再定绝对目标值。

## 8. 实施顺序与依赖

- 依赖：任务详情渐进展示引用状态维护计划的 `evidence` 字段与 `completionNote`；缺口清单的一键动作复用其卡片/深链端点；§5 状态摘要依赖瘦身计划的 `getEffectivePhasesForProjectLike`。**排在两份已有计划之后、与一人多岗计划可并行**（呈现层纯前端；§5 增加一个服务端摘要查询+一致性测试，不动权限模型）。
- **建议实施次序：先 §5 后 §1-§4 与 §6**——焦点三卡、缺口清单、进度显示、"我的工作"分桶全部消费统一的服务端口径，先立数据层再改页面，避免新页面接旧口径返工。§6 的分桶器与状态维护计划的每日摘要分类共用实现，两者应同一批落地。
- 与 6-25 两份 UI 设计的关系：项目详情五标签收口、任务详情两栏结构**照旧执行**，本设计是其上的默认视图/折叠规则/缺口清单三处增量。

## 9. 未决事项

## 10. 实施结果（2026-07-13）

已落地：
- **§5**：`getProjectProgressSummaries`（db.ts，skipped 双侧剔除、按阶段分组一次 SQL）；portfolio 行与 projects.list 行携带 `progress/phaseProgress`；新端点 `projects.statusSummary`（进度+阶段进度+nextGate 缺口+逾期红线任务）；`projects.progress` 降级为派生缓存（refresh 末尾差异回写）；客户端 `computeOverallProgress/computePhaseProgress` 已删除（`getOverallProgress/getPhaseProgress` 只读摘要）；一致性测试 `server/status-summary-consistency.test.ts`（组合看板=单项目摘要=进度缓存，含 lite 档 verification 阶段）。
- **§4**：GateReadinessChecklist 升级为缺口清单——前置任务缺口行显示 任务名+@责任人/待指派+「去处理→」跳转（经 GateReviewModal 线程到任务列表定位），其余维度照旧分行。
- **§1**：ProjectDashboard 顶部焦点三卡（下一里程碑/前三阻断/待决事项），数据源 statusSummary + workbench 行动项，排版固定、空态"无阻断 ✓"。
- **§2**：项目任务列表默认"我的任务"（员工）/"全员任务"（PM/管理层），我的视图按 逾期>今日到期>可开始>进行中 排序；阶段导航条显示各阶段 done/total 摘要（手风琴头）。
- **§3**：任务详情主栏状态渐进首屏——待开始（前置状态+▶开始，前置未清禁用并注明"不是阻塞"）/进行中（证据口径提示+标记完成）/待审批提示/已完成（完成结论展示）；completionNote 通过 tasks.list 水合。
- **§6**：分桶器 `shared/my-work.ts classifyMyWork` 下沉 shared 层，`workbench.mine` 返回 `buckets`；总览工作台渲染 现在处理/等待别人/仅关注 三桶。

复审整改（2026-07-13 第二轮，用户评审 6 项）：
- 🔴 客户端 `computeGateReadiness` 已删除：Gate 子页阻塞提示、任务详情 Gate 面板（改为复用 GateReadinessChecklist 缺口清单组件）、评审弹窗 blockers 参数全部改读服务端 `gateReviews.readiness`——就绪结论全站单一来源。
- 🔴 "现在处理"改渲染服务端 buckets（旧客户端队列仅作分桶缺失时的回退）；workbench 审核查询扩为 reviewer∪submittedBy，"等待别人"分支可命中；任务标题服务端按项目模板解析。
- 🟠 "我的"排序补依赖判定：逾期→今日到期→可开始（前置已清）→进行中→等待别人；无负责人且无岗位限制的任务不再进"我的"。
- 🟠 statusSummary 补齐 health（与组合看板同源 RAG）/blockerCount/counts；总览页任务计数与"下一 Gate"改读摘要，删除本地重算。
- 🟠 进度缓存改用当前（可能是事务）连接计算，修复事务内写旧值。

第三轮整改（2026-07-13，用户复审 4+1 项）：
- 跨项目"我的工作"补依赖判定：workbench 按项目有效依赖图（含裁剪收缩）计算 depsResolved，classifyMyWork 把前置未清的 todo 分入"等待别人"（等前置：…），可开始桶只收前置已清；配 shared/my-work.test.ts 专项测试。
- 旧客户端队列回退已移除（分桶未返回时显示加载态，不再存在第二套口径）。
- 任务排序日期改 todayShanghai()（UTC 在上海凌晨 0-8 点慢一天）。
- 组合看板任务计数 SQL 与进度摘要同口径（skipped 双侧剔除、done 只算 done），同卡"13% vs 4/25"矛盾消除；一致性测试补断言。
- Gate 任务详情不再重复渲染交付物勾选清单（完成度只在缺口清单一处；裁剪/审核动作面板保留）。
- 性能：getPortfolio 支持单项目 scope，projects.get 与 statusSummary 不再各自全组合扫描。
