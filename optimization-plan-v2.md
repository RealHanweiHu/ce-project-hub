# 珠峰项目管理网站 (CE Project Hub) 深度优化计划

基于对当前系统架构的深入分析，我们制定了以下 5 个核心优化方向，旨在解决最紧迫的性能瓶颈、提升代码可维护性、并增强系统的可观测性与测试覆盖率。

## 1. 列表分页与查询治理（最紧迫）

当前系统在数据量增长时面临直接的性能风险。
* **无分页查询**：`tasks.list`、`issues.list` 和 `projects` 列表当前缺乏 `limit`/`offset` 控制。当单项目任务数量破百后，将直接拖慢首屏加载速度。
* **数据库查询优化**：`getProjectsByUser` 和 `getProjectsByMember` 当前并行执行后在内存中合并，这不仅消耗应用服务器内存，且效率低下。应将其合并为一条带有 `LEFT JOIN` 和 `UNION` 的 SQL 语句，并为 `creatorId` 添加索引以加速查询。
* **前端分页支持**：为所有列表接口添加 cursor 分页和服务端排序。前端配合使用 React Query 的 `useInfiniteQuery`，实现平滑的滚动加载或分页体验。

## 2. 前端代码拆分（用户体验直接感知）

当前前端所有 85 个 `.tsx` 文件被打包在一个单一的 bundle 中，导致首屏加载时间过长。
* **按需加载 (Lazy Loading)**：对于 `admin`（后台管理）、`gantt`（甘特图视图）和 `SOP`（SOP 流程库）这种低频但重量级的模块，应使用 `React.lazy()` 进行代码分割。预计实施后，首屏 JS 体积可下降 40% 以上。
* **第三方库拆分**：在 Vite 配置中添加 `manualChunks`，将 `Radix UI`、`date-fns`、`recharts` 等大型第三方依赖拆分为独立的 vendor chunk，以便更好地利用浏览器缓存。

## 3. tRPC 缓存策略统一

当前前端的请求策略存在重复拉取和缓存失效不精确的问题。
* **查询策略优化**：`useProjectData` 目前一次并发 7 个 query，每个查询的 `staleTime` 仅为 5s，导致重复刷新。应建立层级化的 query key（例如 `['project', id, 'tasks']`），配合 `prefetch` 和 `invalidate` 实现精确的缓存失效，并将 `gcTime`（垃圾回收时间）调整至 5 分钟。
* **服务端内存缓存**：对于高频读取的数据（如项目元信息、SOP 模板），可在服务端引入内存缓存（设置 30s 的 TTL），以显著降低数据库的查询压力。

## 4. 可观测性补齐

系统的稳定运行需要完善的监控和日志体系。
* **审计日志保障**：当前活动日志写入数据库失败时仅使用 `console.warn`，这会导致关键的审计信息丢失。应引入 Sentry 或类似的日志聚合工具，对于关键写入失败必须触发告警（Alert）而非静默吞没（Swallow）。
* **全链路追踪**：在系统中添加 Request ID，使其贯穿 tRPC 到数据库层，以便在出现问题时能够快速追踪和定位排查。

## 5. 测试覆盖前移

当前系统的 45 个测试用例集中在服务端的 smoke test，前端测试覆盖率为零。
* **权限组件单测**：优先为权限相关的组件补充单元测试。权限漏洞是系统最大的风险点，必须通过单测保证其逻辑的严密性。
* **E2E 测试覆盖**：引入 Playwright，覆盖系统的核心业务路径（Golden Path）：从“创建项目”到“指派任务”，再到“Gate 评审”，最后“项目归档”。确保核心流程的端到端稳定性。
